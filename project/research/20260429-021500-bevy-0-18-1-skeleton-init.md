## Question 2 — `dynamic_linking` on Bevy 0.18 (and macOS caveats)

**Confidence: HIGH** on feature name and existence; **MEDIUM-HIGH** on macOS caveats (well-known but specifics evolve with Apple toolchain releases).

### Feature name and shape

The feature name is **`dynamic_linking`** (snake_case, on the `bevy` umbrella crate). The standard wiring is exactly what the roadmap shows:

```toml
[features]
dev = ["bevy/dynamic_linking"]
```

Run with `cargo run --features dev`. Build a release binary with plain `cargo build --release` (no `dev` feature, so the dylib path is not used).

This feature has been stable since ~Bevy 0.5 and is documented in Bevy's own quickstart guide. Verified across multiple versions; nothing in the 0.18 release notes (per the existing research doc) flagged it as renamed or removed.

### macOS caveats — the things that bite

1. **Linker choice matters more than the feature.** `dynamic_linking` produces a dylib for the bevy crate; *linking* to that dylib is what gets fast. macOS's default linker (`ld`) is OK but not great. The well-known speedup is to switch the linker:

   - `lld` (LLVM linker) historically had macOS quirks; the recent solution is **`mold` (sold)** or **`ld64.lld`** (LLVM's macOS-specific lld). The current 2025-era recommendation is the `mold` fork called `sold` (or just `lld` on Apple Silicon with recent toolchains).
   - This is configured in `.cargo/config.toml`:
     ```toml
     [target.x86_64-apple-darwin]
     rustflags = ["-C", "link-arg=-fuse-ld=/path/to/sold"]

     [target.aarch64-apple-darwin]
     rustflags = ["-C", "link-arg=-fuse-ld=/path/to/sold"]
     ```
   - **For Feature #1 specifically: do not block on this.** The `dynamic_linking` feature alone provides most of the win. Linker tuning is a separate, optional follow-up. Feature #1 is empty-plugin scaffolding; it's not the right time to tune linkers.

2. **Library lookup at runtime.** Dynamically linked Bevy means the produced binary needs to find `libbevy_dylib.dylib` at runtime. `cargo run --features dev` handles this automatically (it sets `DYLD_LIBRARY_PATH` to `target/debug/`). If a developer copies the binary out of `target/debug/`, it will fail to launch. **Action:** never ship a binary with `dynamic_linking` enabled — the existing research doc's pitfall #1 in roadmap (line 95) calls this out: *"`dynamic_linking` cannot be used in release builds and complicates CI; you have to remember to disable it for shipping artifacts."*

3. **Codesigning / Gatekeeper.** macOS 14+ may complain about an unsigned dylib on first launch. Allowing the binary in System Settings → Privacy & Security solves this. This is a one-time annoyance, not a blocker for daily dev.

4. **Apple Silicon vs Intel.** Both work. No special configuration beyond the standard target triple.

### Workspace-feature opt-in pattern

The roadmap's pattern is correct:

```toml
[features]
default = []
dev = ["bevy/dynamic_linking"]
```

This way `cargo run` (no flags) is a release-shaped dev build (no dylib, faster startup), and `cargo run --features dev` is the daily fast-iteration mode. Document both in a `justfile` or `README` so the user remembers.

### Known issues (HIGH-MEDIUM confidence)

- **Bug class:** `dynamic_linking` occasionally regresses on Bevy point releases. The Bevy GitHub issue tracker has a recurring search term: `is:issue dynamic_linking`. None of the issues I'm aware of from training data are 0.18.1-specific blockers, but the planner should be ready to disable the `dev` feature temporarily if a clean build fails immediately after `cargo update`.
- **Counterargument to hot-reloading expectations:** `dynamic_linking` is a *compile-time* speed-up, not hot-reloading. It does not let you change a system function and see it in a running game without a restart. For hot-reloading, see `bevy_simple_subsecond_system` or `dexterous_developer` — out of scope for Feature #1.

## Question 3 — macOS / Darwin windowing specifics

**Confidence: HIGH** — winit's platform abstraction has been stable for years.

### The short answer

**On macOS, `bevy_winit` provides the windowing layer via Cocoa (AppKit/UIKit), automatically, with no extra feature flag required.** There is no `cocoa` feature flag you need to enable, and there is no equivalent of `x11`/`wayland` for Mac.

- `x11` and `wayland` are **Linux-only** features. They expose conditional code paths in winit that only compile on Linux targets.
- Listing `x11` on macOS is **harmless but noisy**: Cargo will accept it, the code paths are gated by `#[cfg(target_os = "linux")]` and never compile on Mac, but the feature still pulls in the `x11-dl` crate to the dependency graph (small but unnecessary).
- The clean approach for cross-platform support is target-conditional dependencies:

  ```toml
  [dependencies]
  bevy = { version = "=0.18.1", default-features = false, features = [
      "3d", "png", "ktx2", "zstd", "bevy_text",
  ] }

  # Linux-only: enable x11 windowing
  [target.'cfg(target_os = "linux")'.dependencies]
  bevy = { version = "=0.18.1", features = ["x11"] }
  ```

  But for a macOS-first solo project at Feature #1 stage, **just omit `x11` entirely**. Add this target-cfg pattern when Linux CI is wired up — not now.

### Verification

The winit documentation and Bevy's own examples confirm that no platform-specific feature flag is needed for macOS — winit detects the OS at compile time and uses Cocoa. Bevy's official examples directory (e.g. `examples/window/`) compile on macOS without any additional flags. (Source: existing research doc cites the bevy.org examples site as a HIGH-confidence reference.)

### Recommendation

For Feature #1's `Cargo.toml`:
- **Drop `x11` from the feature list.** It's Linux-only.
- **Drop `wayland` discussions.** Not relevant on macOS.
- **No `cocoa` feature is needed.**
- The window will open on macOS using Cocoa transparently via `bevy_winit`.

If the user later wants to support Linux via CI, the target-cfg pattern above is the standard approach.

## Question 4 — Plugin trait API on Bevy 0.18

**Confidence: HIGH** — Plugin trait is one of the most stable APIs in Bevy and has not changed in shape for many versions.

### Minimal Plugin skeleton

```rust
use bevy::prelude::*;

pub struct DungeonPlugin;

impl Plugin for DungeonPlugin {
    fn build(&self, _app: &mut App) {
        // Empty for Feature #1 — systems and resources land in later features
    }
}
```

That's the entire signature you need. The other Plugin trait methods (`name`, `is_unique`, `ready`, `finish`, `cleanup`) all have sensible defaults and you do not need to override any of them for an empty plugin.

### Is there a `#[derive(Plugin)]`?

**No.** As of training-data knowledge through January 2026, Bevy does not ship a `derive(Plugin)` macro in core. Manual `impl Plugin for X` is the standard pattern, and is what the existing research doc, Bevy examples, and every community Bevy project use. **HIGH confidence** — this is one of the longest-stable patterns in Bevy.

(There are third-party crates that auto-implement Plugin via attribute macros for convenience, but they're niche and not the standard. Don't add them in Feature #1.)

### Plugin trait surface (full signature, for reference)

```rust
pub trait Plugin: Any + Send + Sync {
    fn build(&self, app: &mut App);                    // REQUIRED
    fn ready(&self, _app: &App) -> bool { true }       // optional
    fn finish(&self, _app: &mut App) {}                // optional
    fn cleanup(&self, _app: &mut App) {}               // optional
    fn name(&self) -> &str { std::any::type_name::<Self>() }  // optional
    fn is_unique(&self) -> bool { true }               // optional
}
```

For empty plugin scaffolding: only `build` matters. Leave `_app` underscore-prefixed to avoid unused-variable warnings.

### Idiomatic file shape for Feature #1

`src/plugins/dungeon/mod.rs`:

```rust
use bevy::prelude::*;

pub struct DungeonPlugin;

impl Plugin for DungeonPlugin {
    fn build(&self, _app: &mut App) {}
}
```

Repeat for `CombatPlugin`, `PartyPlugin`, `TownPlugin`, `UiPlugin`, `AudioPlugin`, `SavePlugin` — one per `src/plugins/<name>/mod.rs`. ~5 lines each.

### Counterargument check

> "Should the plugins use a single `Plugin` macro from a third-party crate to reduce boilerplate?"

**No, not at Feature #1.** The boilerplate is 5 lines per plugin. Adding a macro dependency to save lines you'll write seven times is a textbook over-engineering anti-pattern. Stay vanilla.

> "Should I use `App::add_plugin(MyPlugin)` (singular) instead?"

**Use the plural form.** `add_plugin` (singular) was the older API; `add_plugins` (plural, accepting either a single plugin or a tuple) is the standard since Bevy ~0.11 and is what 0.18 uses. See Question 5.

## Question 5 — `App::new().add_plugins(tuple)` API on 0.18

**Confidence: HIGH** — this API has been stable since Bevy 0.11 and there is nothing in the 0.18 release notes (per existing research) flagging a change.

### Confirmed shape

```rust
use bevy::prelude::*;
use druum::plugins::{
    audio::AudioPlugin,
    combat::CombatPlugin,
    dungeon::DungeonPlugin,
    party::PartyPlugin,
    save::SavePlugin,
    town::TownPlugin,
    ui::UiPlugin,
};

fn main() {
    App::new()
        .add_plugins((
            DefaultPlugins,
            DungeonPlugin,
            CombatPlugin,
            PartyPlugin,
            TownPlugin,
            UiPlugin,
            AudioPlugin,
            SavePlugin,
        ))
        .run();
}
```

Key facts the planner needs:

1. **`add_plugins` accepts both a single plugin and a tuple.** The tuple form is what you want here — it's the canonical idiom.
2. **Tuples can be nested for organization.** `add_plugins((DefaultPlugins, (DungeonPlugin, CombatPlugin), (UiPlugin, AudioPlugin)))` works. The flat form above is fine for seven entries; nested is useful at 15+.
3. **`DefaultPlugins` is a `PluginGroup`, not a single plugin** — but `add_plugins` accepts both `Plugin` and `PluginGroup` as tuple elements. So you can mix them freely as shown above.
4. **`DefaultPlugins` is in the prelude** (`use bevy::prelude::*;` is sufficient). Confirmed across all recent Bevy versions; HIGH confidence.
5. **Multiple `add_plugins` calls compose.** `App::new().add_plugins(DefaultPlugins).add_plugins((DungeonPlugin, CombatPlugin))...` works identically. Useful when you want to split plugin groups across modules.

### Tuple-size limit

Rust tuple traits are typically implemented for tuples up to 16-20 elements. With seven plugins + `DefaultPlugins`, you are well under the limit. If the project ever crosses ~15 plugins, nest tuples or split into multiple `add_plugins` calls.

### What changed across versions (context, not a 0.18 issue)

- Pre-0.11: `add_plugin(MyPlugin)` (singular), `add_plugins(MyPluginGroup)` (plural for groups only). Two functions, easy to confuse.
- 0.11+: Unified into `add_plugins(...)` accepting both. Singular `add_plugin` deprecated/removed.
- 0.18: No further changes to this API.

So **the planner can copy the example above verbatim** and expect it to work on 0.18.1.

### Counterargument check

> "Should I use `MinimalPlugins` instead of `DefaultPlugins` for the smoke test?"

**No.** `MinimalPlugins` doesn't include a window, renderer, or input handlers — using it for a smoke test won't open a window. The roadmap's success criterion is "opens a black window and exits cleanly on Cmd+Q" which requires `DefaultPlugins`.

## Question 6 — Bevy 0.18.1 MSRV and `rust-toolchain.toml`

**Confidence: MEDIUM** — I cannot directly read the 0.18.1 manifest's `rust-version` field. Reasoning from Rust release cadence and Bevy's MSRV-tracking habit is below.

### Best estimate

Bevy historically tracks within ~2 minor versions of stable Rust. 0.17.x had MSRV around 1.83 (released Nov 2024). 0.18.1 was released **2026-03-04** (per the existing research doc, line 9). By that date, Rust 1.85 (Feb 2026) would be the most recent stable; Rust 1.86 may or may not have been out depending on exact release timing.

**Estimate:** Bevy 0.18.1's MSRV is **1.85 or 1.86**.

### Authoritative verification

Either of these gets the exact answer in seconds:

```bash
# Option A — read the published manifest directly:
curl -s https://raw.githubusercontent.com/bevyengine/bevy/v0.18.1/Cargo.toml | grep '^rust-version'

# Option B — after first compile, ask cargo:
cargo metadata --format-version=1 | jq '.packages[] | select(.name=="bevy") | .rust_version'
```

### Recommended `rust-toolchain.toml`

Pinning to a specific stable version makes CI reproducible and prevents surprise compiler-version drift. The roadmap's "1.85+" suggestion is reasonable.

```toml
# rust-toolchain.toml
[toolchain]
channel = "1.85.0"
components = ["rustfmt", "clippy"]
profile = "minimal"
```

Use `1.85.0` (or whichever exact version the verification step above returns — bump to `1.86.0` if Bevy's MSRV is higher). `profile = "minimal"` keeps installs fast; `components` ensures `cargo fmt` and `cargo clippy` work without a separate `rustup component add` step.

**Why pin to a specific patch?** You want CI and your machine to match exactly. `channel = "stable"` floats and produces non-reproducible diagnostics across machines. `channel = "1.85"` (no patch) is acceptable but slightly looser.

### Counterargument check

> "Should I just use `channel = "stable"` and let rustup pick the latest?"

For a solo greenfield project, `stable` is tempting and fine. The downside: if you walk away for a month and come back after `rustup update`, you may inherit new lints that flag your code as warnings. Pinning protects against this. Tradeoff: small. I lean pin.

> "Why not use a nightly channel for `cargo build -Zbuild-std` or other speedups?"

Bevy compiles fine on stable, doesn't require nightly features, and nightly's instability would conflict with the roadmap's Pitfall #1 mitigation goal (stability over leading-edge). Stick with stable.

## Question 7 — Smoke test expectations (`cargo run --features dev`)

**Confidence: HIGH** on the broad behavior; **MEDIUM** on the exact default clear color and exact log strings (which can change between Bevy versions).

### What you should see

Running `cargo run --features dev` with `DefaultPlugins` + seven empty plugins:

1. **Compile.** First build: 30-90 seconds. Subsequent incremental builds with `dynamic_linking`: 5-8 seconds. (Source: existing research doc line 1118-1119, HIGH confidence.)
2. **A window opens.** Title likely defaults to the binary name (`druum` or `Bevy App`) depending on Bevy 0.18 defaults. Size defaults to ~1280x720 or similar — Bevy's default `Window` is set in `WindowPlugin`.
3. **The window's content is the default clear color.** With no camera, nothing is rendered into the window's framebuffer. The framebuffer is cleared each frame to the default `ClearColor` resource value. **Without explicit configuration**, this is typically a dark gray (~`Color::srgb(0.13, 0.13, 0.13)`) or whatever the renderer's default `ClearColor` resource initializes to. Bevy 0.13 introduced a "Bevy purple" gradient, but I cannot confirm which default 0.18 ships. **Expect: blank window, single solid color.**
4. **Logs print to terminal.** You will likely see a tracing/log message similar to "no camera entity found" or "no camera with target Window(...)" — not an error, just a warning that nothing is being rendered. Exact wording varies by Bevy version.
5. **Cmd+Q (macOS) closes the window cleanly.** `DefaultPlugins` includes `WindowPlugin` which wires up the platform's standard quit shortcut. Process exits with code 0.

### What you will NOT see (and shouldn't worry about)

- **No 3D scene, no objects, no anything.** That's correct — the seven plugins are empty stubs.
- **No FPS counter or debug overlay.** Those would require enabling `bevy_dev_tools` or adding `FrameTimeDiagnosticsPlugin`. Out of scope for Feature #1.
- **No flicker or visible artifacts.** A solid color is the expected steady state.

### Is a camera required for the window to appear?

**No.** The window appears regardless because `WindowPlugin` is responsible for window creation, independent of cameras. Cameras are about *rendering into* the window's framebuffer.

What happens *without* a camera:
- The framebuffer is still cleared each frame (by the renderer's clear pass) to `ClearColor`.
- No render commands are issued because there's no view. So the window shows the clear color and nothing else.
- Bevy emits a logging warning. This is a **soft warning, not an error**. It does not crash the app.

The roadmap's instruction (line 134: *"Resist the urge to also wire up egui, audio, and input here"*) explicitly extends to cameras. **Do not add a camera in Feature #1.** The smoke test is meant to verify that the window opens and process exits cleanly — the warning about no camera is the expected, healthy signal.

### Acceptance criteria for Feature #1's smoke test

The roadmap (line 131) states:
> "Confirm `cargo run --features dev` opens a black window and exits cleanly on Cmd+Q."

Refining this slightly based on the analysis above:

- [ ] `cargo run --features dev` builds and starts a window.
- [ ] Window content is a solid color (likely dark gray; "black" in the roadmap is approximate).
- [ ] Console emits a "no camera" log warning (or similar). This is **expected and not a failure**.
- [ ] Cmd+Q closes the window. The process exits with code 0.
- [ ] `cargo run` (no `--features dev`) also works — verifies the project compiles without dynamic linking.
- [ ] `cargo build --release` succeeds — verifies the dynamic_linking feature is properly gated and doesn't bleed into release builds.

### Counterargument check

> "If the screen is going to be the wrong color, shouldn't we just add a `ClearColor` resource to make it explicit?"

This is reasonable, but the roadmap is ruthless about scope. Setting `ClearColor` is one extra line of code you'd write in `main.rs`, which is fine, but it crosses into "configuration-of-future-systems" territory. **Recommendation: do not set `ClearColor` in Feature #1.** The point is "does the empty scaffold compile and run." If the window color matters for the smoke test acceptance, document the *observed* color rather than enforcing a specific one.

> "What if the window doesn't appear at all on macOS?"

If a window doesn't appear, the most likely causes are: (a) the Cocoa event loop didn't initialize (rare; means winit/bevy_winit broke), (b) macOS Gatekeeper is holding the binary (check System Settings → Privacy & Security), (c) the binary isn't actually using `DefaultPlugins`. Debug by adding `App::new().add_plugins(DefaultPlugins).run()` with NO custom plugins and seeing if that produces a window. If that fails, the bevy install is broken; not a Feature #1 logic issue.
