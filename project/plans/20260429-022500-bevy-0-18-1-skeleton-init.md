## Goal

Stand up a compilable Bevy 0.18.1 project skeleton at `/Users/nousunio/Repos/Learnings/claude-code/druum/` with seven empty `Plugin` impls (`DungeonPlugin`, `CombatPlugin`, `PartyPlugin`, `TownPlugin`, `UiPlugin`, `AudioPlugin`, `SavePlugin`), a `dev` feature flag for `dynamic_linking`, dev-profile tuning, pinned Rust toolchain, and a `justfile` with `check` / `clippy` targets. Smoke test: `cargo run --features dev` opens a window and exits cleanly on Cmd+Q.

## Approach

Bevy 0.18 reorganized its umbrella crate into feature-gated subcrates (`3d`, `ui`, etc.), so the standard pattern is `default-features = false` plus a cherry-picked feature list. The research recommends using only the `3d` umbrella plus image-format features (`png`, `ktx2`, `zstd`) and `bevy_text`, then letting Cargo report errors on the first build if any of `bevy_winit`/`bevy_pbr`/`bevy_asset` are not transitively pulled in by `3d`. macOS-only dev means dropping `x11` (Linux-only) entirely. Dev-mode iteration speed comes from two layers: `[profile.dev.package."*"] opt-level = 2` (deps run fast even when our crate is at `opt-level = 1`) plus the `dev` feature flag enabling `bevy/dynamic_linking` (links Bevy as a dylib for ~5-8s incremental rebuilds vs. 30s+ static).

The plugin layout is the seven module flat tree the roadmap specifies. Each plugin lives at `src/plugins/<name>/mod.rs` with a 5-line empty `Plugin` impl. `src/main.rs` wires them via `App::new().add_plugins((DefaultPlugins, ...)).run()`. No camera, no `ClearColor`, no state machine ‚Äî Feature #1 is intentionally tiny per the roadmap's "Additional Notes" line, leaving room for Features #2 (state machine), #3 (asset pipeline), #5 (input), and #6 (audio) to land cleanly later.

Two MEDIUM-confidence items from the research need on-the-spot verification: (1) the exact spellings of bevy 0.18.1's feature flag names ‚Äî solved by a 30-second `curl` of `crates/bevy_internal/Cargo.toml` at tag v0.18.1 BEFORE locking `Cargo.toml`, and (2) the actual MSRV ‚Äî solved by a parallel `curl` of the workspace root `Cargo.toml`. Both verifications happen in Step 1 with a documented fallback (use the research's recommended starter values if curl fails for any reason).

## Critical

- Bevy version is pinned to **`=0.18.1`** (with `=` operator) ‚Äî not `^0.18.1` and not `0.18.1`. An accidental `cargo update` to 0.18.2 or 0.19 must require deliberate edit, per research Pitfall 1.
- The `dev` feature must be **opt-in only** (not in `default`). `cargo build --release` must work without `dynamic_linking` bleeding in ‚Äî research Q2 caveat #2: never ship with `dynamic_linking` enabled.
- **Do not add a camera, `ClearColor` resource, state machine, asset loader, input system, audio plugin contents, or any 3rd-party crates beyond Bevy itself.** All seven `Plugin` impls remain empty stubs. Roadmap line 134: "Resist the urge to also wire up egui, audio, and input here."
- **Do not run `cargo init`.** The cwd already exists with non-empty contents (`research/`, `project/`, `.claude/`); `cargo init` would either reject the directory or pollute it. Create `Cargo.toml` and `src/main.rs` directly via Write.
- **Drop `x11` from the feature list.** It's Linux-only. macOS uses Cocoa transparently via `bevy_winit` (research Q3, HIGH confidence).

## Security

**Known vulnerabilities:** No known CVEs affecting `bevy = 0.18.1` as of 2026-04-29 (research date). The `=0.18.1` pin is itself a security posture: it prevents an opportunistic `cargo update` from pulling a yet-unreleased compromised point release without the team noticing. Re-evaluate when intentionally bumping to 0.18.2/0.19.

**Architectural risks:** Feature #1 introduces no network I/O, no untrusted-input parsing, no asset loading at runtime, and no plugin contents ‚Äî there are no trust boundaries to harden. Two architectural guards already in this plan:

- The `dynamic_linking` feature is gated behind a non-default `dev` feature so a release build cannot accidentally ship a binary with a runtime dylib dependency (research Q2 caveat #2: shipping a `dynamic_linking` binary fails to launch outside `target/debug/`).
- The Bevy version is pinned with `=0.18.1` (exact-match operator), not `^0.18.1`, so dependency drift requires a deliberate edit.
