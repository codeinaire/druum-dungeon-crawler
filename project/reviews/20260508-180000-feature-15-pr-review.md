### [MEDIUM] `UseItem` heal path missing `check_dead_and_apply` after HP write

**File:** `src/plugins/combat/turn_manager.rs:556-560`

**Issue:** The `UseItem` consumable branch writes `current_hp = current_hp.saturating_add(heal).min(max_hp)` (line 559) but does NOT call `check_dead_and_apply` afterward. The plan's critical constraint states "After every HP write, call `check_dead_and_apply`." While a heal can only raise HP, it cannot produce a zero-HP write that would trigger the Dead path — so this specific omission is not a gameplay bug today. However, the constraint exists because future modifications to the UseItem branch (e.g., items that apply a damage component, drain HP, or grant a null heal to a dead entity with 0 max_hp) could silently violate the invariant. The contract is stated as absolute, not conditional on the sign of the HP delta.

More concretely: if `max_hp == 0` (a degenerate default-constructed entity that somehow passed the `is_alive_entity` gate), `heal = 0`, `current_hp` stays 0, and `check_dead_and_apply` is never called. The Dead event is never emitted for that entity. This is a v1 edge case only reachable via test-fixture misconstruction, but it's a latent hole in the invariant.

**Fix:** Add `check_dead_and_apply` after the HP write, consistent with the Attack arm:

```rust
// After line 559 (the current_hp update):
if let Ok(mut d) = derived_mut.get_mut(action.actor) {
    let heal = d.max_hp / 4;
    d.current_hp = d.current_hp.saturating_add(heal).min(d.max_hp);
    // Maintain the invariant: check after EVERY HP write.
    check_dead_and_apply(action.actor, &d, &mut apply_status);
}
```

### [LOW] UseItem heal logs "drinks X!" even when heal amount rounds to zero

**File:** `src/plugins/combat/turn_manager.rs:562`

**Issue:** Integer division `d.max_hp / 4` truncates to 0 when `max_hp < 4`. The consumable log entry `"{} drinks {}!"` fires unconditionally before the heal is applied. A party member with `max_hp = 3` would consume their item (removing it from inventory) and see the drink message with no visible HP change. This wastes the item.

This is a cosmetic/UX gap in v1 (max_hp < 4 is not a normal stat range at level 1 with proper stats), but it would be trivially avoidable.

**Suggested fix:** Early-exit if heal amount is 0:

```rust
let heal = d.max_hp / 4;
if heal == 0 {
    combat_log.push(
        format!("{} fumbles with {} (no effect).", actor_name, display_name),
        turn,
    );
    continue;
}
// ... rest of consume + log
```

## Required follow-up commit

**Do not merge this PR without the follow-up commit.** Five bugs were found and fixed during manual UI smoke testing after the PR was opened. These fixes are currently uncommitted on `gitbutler/workspace` and must land before merge:

1. **Combat exit-on-entry bug** — `despawn_dungeon_entities` (`dungeon/mod.rs:554`) was unconditionally despawning the party on `OnExit(Dungeon)`, including when transitioning to Combat. Combined with the now-resolved LOW-2 vacuous defeat, this caused immediate `Combat → GameOver` on F9. Fix: conditional party despawn, idempotent `spawn_party_and_camera`, `!party.is_empty()` guard in `check_victory_defeat_flee` (this last item is already in the PR per the re-verification above — the dungeon-side guard is the remaining piece).

2. **Dev party spawned with zero stats** — `spawn_default_debug_party` used `..Default::default()` giving `BaseStats::ZERO` and `DerivedStats { current_hp: 0, ... }` — all members had hp=0, triggering immediate defeat. Fix: explicit BaseStats (VIT=12 → 17 max_hp) + `derive_stats(...)` call.

3. **Main action menu was un-navigable** (hardcode: Confirm always opened Attack). Fix is already in the PR (`main_cursor` navigation landed in `25cfad4`/`dcf3d51`).

4. **Status display only showed effect count** (`[1]` instead of `[DefenseUp(2t ×0.5)]`). Fix with `format_status_effects()` is already in the PR.

5. **Camera order ambiguity warning** when F9 cycles `Combat → GameOver → Loading`. Fix: `cleanup_party_after_combat` system on `OnExit(Combat)` that despawns party if destination is not Dungeon.
