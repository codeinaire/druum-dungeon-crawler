Feature #12 lands the data layer for items, inventory, and equipment on top of #11's frozen character schema. Zero new deps (Cargo.toml byte-unchanged — same cleanest-ship signal as #7/#8/#9/#11). UI is explicitly punted to #25; save/load to #23. Several decisions here will shape downstream features:

- **Hybrid model is the architecture: `Equipment` = `Option<Handle<ItemAsset>>` (FROM #11, unchanged), `Inventory` = `Vec<Entity>` per character (NEW in #12), each entity carries `ItemInstance(Handle<ItemAsset>)`.** **Why:** #11 already shipped `Equipment` with `Handle` (Decision 3 — locked at `character.rs:209-219`); reverting requires reopening that decision. The doc-comment promise at `character.rs:204-205` explicitly names #12 as the home of the `ItemInstance` entity model. The Hybrid is the only architecture that satisfies both prior commitments. **How to apply:** when planning #15 (combat), #21 (loot), #25 (UI), the model is set — equipped items are handles (look up `Assets<ItemAsset>` for stats), in-bag items are entities (carry `ItemInstance` plus future per-instance state). #18 (shop) and #21 (loot) are the "second consumer" that may layer on `Enchantment(u8)` / `Durability(u32)` components on the inventory entities — DO NOT pre-create those in #12.

- **`derive_stats` is FROZEN — the recompute system owns the flatten step.** `derive_stats(base, equip_stats: &[ItemStatBlock], status, level)` is pure (per `character.rs:323`). #12's `recompute_derived_stats_on_equipment_change` system reads each `Option<Handle<ItemAsset>>` slot, looks up `Res<Assets<ItemAsset>>`, builds `Vec<ItemStatBlock>`, calls `derive_stats`. **Why:** #11 deliberately put the flatten step in the caller for testability. Reverting `derive_stats`'s signature breaks all #11 unit tests. **How to apply:** any future stat-recompute callsite (#14 level-up, #15 combat status changes) follows the same pattern — flatten in the caller, call `derive_stats` pure. The "caller-clamp" contract (`current = current.min(new_max)`) lives at the recompute site, NOT inside `derive_stats`. Refactor to a shared `apply_derived_stats_with_clamp(...)` helper only when the second consumer arrives.

- **`EquipmentChangedEvent` is `#[derive(Message)]` (NOT `Event`); registered with `app.add_message::<...>()`; consumed via `MessageReader<...>`.** Bevy 0.18 family rename. The `...Event` suffix is kept for genre familiarity (matches `MovedEvent` precedent at `dungeon/mod.rs:192-197`). **Why:** Bevy 0.18 renamed the `Event` family to `Message`; using `Event` derives compiles but uses the legacy API path. **How to apply:** Verification gate `rg 'derive\(.*\bEvent\b' src/plugins/party/inventory.rs` must return ZERO matches. Same rule for any future cross-system signaling type (#15 `DamageDealt`, #21 `LootDropped`, #14 `LevelUp`). The grep is now part of the standard verification gate for any new message type.

- **`PartyMemberBundle` is FROZEN — `Inventory::default()` is inserted on a SECOND CALL after `commands.spawn(PartyMemberBundle{...})`, NOT added to the bundle.** `character.rs` is FROZEN post-#11 (per the file-level doc comment). The only edit to wire `Inventory` onto debug party members is in `spawn_default_debug_party` at `mod.rs:62`: append `.insert(Inventory::default())` to each spawn call. **Why:** modifying `PartyMemberBundle` would touch frozen `character.rs`. The chained `.insert(...)` is the standard Bevy idiom for layering components onto a bundle-spawned entity without modifying the bundle definition. **How to apply:** any future feature that adds a new component to party members (#15 `CombatStance`, #19 `CharacterPortrait`, #25 `UiPreference`) follows the same `.insert(...)` chain pattern. Do NOT extend `PartyMemberBundle`.

- **`assets/items/core.items.ron` is the locked path — NOT `assets/items/items.ron` (roadmap line 659 is wrong).** The loader at `loading/mod.rs:33` expects `items/core.items.ron`. **Why:** #3 locked the path; the roadmap was authored before #3. Same trap as #11 Pitfall 2 (`classes/core.classes.ron`). **How to apply:** any future `*.ron` content file follows the `core.<domain>.ron` naming convention. Do NOT create a parallel `items.ron` file. Do NOT edit `loading/mod.rs` — it's FROZEN post-#3.

- **`EquipSlot::None` is a sentinel for non-equippable items; `equip_item` validates BOTH the caller-side `slot == None` AND the asset-side `slot == None || kind == Consumable | KeyItem`.** Defense-in-depth: a `Consumable` with a wrongly-authored `slot: Weapon` is rejected by the kind check; a `Weapon` with `slot: None` is rejected by the slot check. Both rejections return `EquipError::ItemHasNoSlot`. **Why:** RON content is the trust boundary; validation must catch bad authoring AND bad caller intent. **How to apply:** any future fallible API on user-authored data (loot tables in #21, shop inventory in #18) follows the same dual-check pattern.

- **`equip_item` / `unequip_item` / `give_item` are PUBLIC HELPER FUNCTIONS (free fns), not Bevy systems.** They take `&mut Commands`, queries, and a `MessageWriter` as parameters. The Bevy system that wraps them maps user input or scripted events to helper calls. **Why:** Bevy systems consume only their parameter list; helpers compose with future callers (#21 loot, #18 shop, #25 UI). The system itself looks like `for req in requests.read() { equip_item(...) }`. **How to apply:** for #15 combat (item-use during combat), #21 loot (chest-give), #25 UI (drag-equip), the helper functions are the public API. Do NOT design those features to call into `equip_item` from a different system shape — `&mut Commands + queries + MessageWriter` is the contract.

- **`Result<(), EquipError>` is the validation return shape — system logs `warn!` on `Err`.** Idiomatic Rust. UI in #25 will eventually consume the `EquipError` enum to render tooltips ("can't equip dagger in armor slot"). **Why:** richer than `bool` (`DungeonFloor::can_move`), composes with #25 UI. **How to apply:** future fallible APIs in this project use `Result<T, E>` with a small `enum E { Variant1, Variant2, ... }` shape; avoid panics, avoid silent no-ops. Logging at the wrapper-system level (NOT inside the helper) is the standard.

- **`stackable: bool` is declared on `ItemAsset` but NO system reads it in v1.** Roadmap line 681 punts. Each potion is a unique entity. **Why:** schema forward-compat without committing to a stack-merge model. The roadmap explicitly punts. **How to apply:** post-v1, when stackable lands (probably alongside #21 loot tables when "5x healing potion" drops first appear), the implementer chooses between (a) `StackableItem(u32)` component on the entity (one entity = N items) OR (b) UI-only deduplication (N entities, one icon with count badge). #12's Inventory(Vec<Entity>) supports both options without redesign.

- **Auto-resolve D1 + D6 (Equipment shape) from #11's already-shipped code.** D1 (`Entity` vs `Handle`): #11 ships `Handle` — locked. D6 (reverse #11 Decision 3): only surface if user asks to reverse. **Why:** reopening #11 Decision 3 is a 200+ LOC redesign with `MapEntities` plumbing that #11 explicitly chose to avoid. **How to apply:** any future plan that touches `Equipment` confirms the shape with #11's `character.rs:209-219` first; the shape is locked.

- **D2 (Inventory shape) is RECOMMENDED `Vec<Entity>` per character but is genuinely Category B if user wants the simpler `Vec<Handle>` model.** The fallback (Option B) drops `ItemInstance`, drops the entity model, costs ~50 LOC less, but loses path to per-instance enchantment. **Why:** the doc-comment promise at `character.rs:204-205` favors the entity model, but the user can override. **How to apply:** if user picks B, the plan-of-record adjusts: drop `ItemInstance`, `Inventory(Vec<Handle<ItemAsset>>)`, equip path becomes a simple Vec ↔ slot move. ~3 fewer Layer-2 tests.

- **D4 (item count: 8 vs 12) is genuine designer balance.** 8 is recommended (3 weapons, 2 armor, 1 shield, 1 consumable, 1 key item — covers 5 of 9 ItemKinds). 12 adds 1 helm/gloves/boots/accessory for full slot coverage in the integration test. **How to apply:** if downstream features (#15 combat, #21 loot, #14 progression) want full slot coverage early, prefer 12. Otherwise default to 8 — Helm/Gloves/Boots/Accessory designs can land alongside #14 race effects when slot bonuses become balance-critical.

- **D5 (debug starter-items system: ship vs skip) — RECOMMEND SKIP for #12.** #21 loot is when items first matter for in-game flow. The Inventory component is still inserted onto debug party members (so #25 UI dev has a consistent model), but no items are pre-populated. **How to apply:** if D5 = A in a future revision, the system follows the SAME idempotence guard as `spawn_default_debug_party` (#12 Pitfall 7) and is `#[cfg(feature = "dev")]`-gated.

- **D8 (placeholder icons): RECOMMEND ImageMagick-generated 32×32 PNGs committed via `scripts/gen_placeholder_icons.sh` — NEW directory `assets/ui/icons/items/`.** The project has NO `assets/ui/` yet (Feature #12 creates the first PNG/icon directory). #25 polishes the icons. **How to apply:** future icon-bearing features (#17 enemy sprites, #19 portraits) follow the `assets/ui/<category>/` convention. The `gen_placeholder_icons.sh` script is committed for repeatability.

- **Test layer split for #12:** Layer 1 (~6 tests, no `App`): `EquipError` returns, `EquipSlot::read/write` round-trip, `ItemAsset`/`ItemDb` RON round-trip. Layer 2 (~3 tests, `MinimalPlugins + AssetPlugin + StatesPlugin + StatePlugin + PartyPlugin`): recompute drives `DerivedStats`, `equip_item` emits message, `give_item` pushes to inventory. Integration (1 test, `tests/item_db_loads.rs`): `RonAssetPlugin` path loads `core.items.ron`. **Total: 10 net new tests, matches roadmap budget +6-10 at line 670.** **How to apply:** #12 Layer 2 introduces a NEW pattern — `App + AssetPlugin` for asset-aware system tests. Future asset-aware features (#15 spell asset effects, #21 loot table parsing) reuse this harness shape. The harness mirrors `audio/mod.rs:145-178` (Layer 2 without InputPlugin) but adds `AssetPlugin::default()`.

- **Atomic commit boundaries (9 phases → ~9 commits, possibly merging Phase 4+5 into one commit).** Phase 2 lands BEFORE Phase 1 in commit order (Phase 1's `data/items.rs` references types defined in Phase 2's `inventory.rs`). The plan documents this in the Phase 1 ordering note. **How to apply:** future features that have circular cross-file references between `data/` and `plugins/` follow the "module-defining file commits first, consumer file commits second" rule. The commit-order constraint is captured in the plan's commit-ordering note at the relevant phase.
