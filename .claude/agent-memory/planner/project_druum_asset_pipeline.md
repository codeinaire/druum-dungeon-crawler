Feature #3 introduces `LoadingPlugin` at `src/plugins/loading/mod.rs` plus stub asset schemas under `src/data/{dungeon,items,enemies,classes,spells}.rs`. Several decisions here will shape downstream features:

- **`LoadingPlugin` owns the `Loading → TitleScreen` exit; `bevy_asset_loader::LoadingState` does the `next.set(...)` internally.** No other plugin sets `NextState<GameState>::TitleScreen`. **Why:** preserves the Feature #2 contract that `GameState::Loading` has no auto-advance from arbitrary places, and avoids racing the asset loader's transition. **How to apply:** Feature #23 (save/load) may RE-enter `Loading` via `next.set(GameState::Loading)` to drive a save-restore — that's allowed because re-entering `Loading` re-triggers `LoadingPlugin`'s `OnEnter` and the loading state restarts. Do NOT add a parallel `next.set(TitleScreen)` in any feature.

- **Data schemas re-export from `src/data/mod.rs` via `pub use` (a deliberate exception to the no-re-export rule).** `crate::data::DungeonFloor` resolves directly. **Why:** asset schemas are referenced from many call sites (loading, dungeon, combat, party, town, save) and the import-clutter cost is real; one tree boundary is the narrow exception. By contrast, `GameState` is deliberately NOT re-exported from `plugins/mod.rs` (Feature #2 convention). **How to apply:** when adding a new typed RON asset family in a later feature, follow the pattern: `src/data/<name>.rs` defines the struct + round-trip test, `src/data/mod.rs` adds both `pub mod <name>;` and `pub use <name>::Type;`. Do NOT re-export plugins, run conditions, or systems — only data schemas.

- **`assets/` directory layout: one folder per typed extension.** `dungeons/*.dungeon.ron`, `items/*.items.ron`, `enemies/*.enemies.ron`, `classes/*.classes.ron`, `spells/*.spells.ron`, `fonts/*.ttf`. Each multi-dot extension dispatches to a unique `Asset` type via `bevy_common_assets::ron::RonAssetPlugin::<T>::new(&["<full.ext>"])` — full extension, no leading dot. **Why:** Bevy's `AssetPath::get_full_extension` strips everything before the first `.`, so `floor_01.dungeon.ron` matches the loader registered as `"dungeon.ron"`. Plain `.ron` would be a single shared loader for all five families and break type dispatch. **How to apply:** when adding a new asset family, pick a unique multi-dot extension. Document in `assets/README.md` (the layout doc lives at the assets root). Do NOT use plain `.ron` for any one type.

- **Hot-reload gating shape: `bevy/file_watcher` in `dev` Cargo feature composition, AND `AssetPlugin { watch_for_changes_override: Some(cfg!(feature = "dev")), ..default() }` in `main.rs`.** Both pieces are required (research §Pitfall 2). The override line uses the `cfg!(...)` MACRO, not a `#[cfg(...)]` ATTRIBUTE, so the line is always compiled in and resolves to `Some(true)` under `--features dev`, `Some(false)` otherwise. **Why:** the override defaults to `None`, which falls back to `cfg!(feature = "watch")`. The macro form keeps `main.rs` uniform across feature sets and ties hot-reload visibly to the `dev` feature. **How to apply:** never set `Some(true)` unconditionally in a release-shipping path — that pulls `notify-debouncer-full` into the binary. Never set `Some(false)` unconditionally — that disables hot-reload even under `--features dev`. The `cfg!()` form is the single correct shape.

- **`AssetCollection` struct must derive BOTH `AssetCollection` AND `Resource`** (from `bevy_asset_loader`). Forgetting `Resource` produces an opaque "trait `Resource` not implemented" error at the `load_collection::<...>()` call site. **How to apply:** every future `#[derive(AssetCollection)]` struct also gets `#[derive(Resource)]`. This is a one-time convention to remember.

- **`AssetEvent<T>` is a `Message` in Bevy 0.18, not an `Event`** — same family-rename trap that bit `StateTransitionEvent` in Feature #2. Reading asset events directly requires `MessageReader<AssetEvent<T>>`. **How to apply:** Feature #3 routes through `bevy_asset_loader` to avoid the trap entirely. If a later feature (e.g., #4 dungeon-on-load processing) needs to react to `Modified` events for hot-reload behavior, it MUST use `MessageReader<AssetEvent<DungeonFloor>>`, not `EventReader<...>`.

- **Loading-screen `Camera2d` is despawned on `OnExit(GameState::Loading)`** alongside the UI tree, via a shared `LoadingScreenRoot` marker component. **Why:** keeps Feature #3 self-contained and avoids cross-feature camera ownership coupling. **How to apply:** subsequent features that need a camera spawn their own (Feature #25 title screen → its own `Camera2d`; Feature #7 dungeon view → `Camera3d`). Do NOT rely on the loading-screen camera persisting.

- **Stub asset types use `#[derive(Asset, Reflect, Serialize, Deserialize, Default, Debug, Clone)]`.** All six derives are load-bearing — `Asset` for the trait + `VisitAssetDependencies` impl, `Reflect` for `TypePath`, serde for the round-trip, `Default` for tests and loader sanity, `Debug` for log output, `Clone` for `bevy_asset_loader`'s populated-resource pathway. **How to apply:** every new typed RON schema follows this exact derive list.

- **Crate-version-lag escalation playbook (NiklasEi crates).** `bevy_common_assets` and `bevy_asset_loader` typically lag Bevy minors by 0-3 weeks. If a future Bevy bump (e.g., to 0.19) requires bumping these too and the upstream lags, do NOT downgrade Bevy — escalate to the user with a one-paragraph note proposing (a) wait for upstream, or (b) hand-roll the lagging crate (~30 LOC for `RonAssetPlugin` replacement, ~50 LOC for `LoadingState` replacement). Same playbook as the `moonshine-save` precedent (roadmap Decision §Resolved #3, 2026-04-29).
