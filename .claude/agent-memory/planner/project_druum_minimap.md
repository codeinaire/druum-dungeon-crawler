Feature #10 adds the auto-map: `MovedEvent`-driven `ExploredCells` resource painted via `bevy_egui` as top-right overlay (Exploring) and full-screen view (Map sub-state). First non-trivial Cargo dep added since Feature #5 (`leafwing-input-manager`). Several decisions here will shape downstream features:

- **Steps A/B/C verification gate is upstream of EVERY code edit.** Same playbook as Feature #3 (`bevy_common_assets`/`bevy_asset_loader`) and Feature #5 (`leafwing-input-manager`): `cargo add bevy_egui --dry-run` resolves the actual Bevy 0.18-compatible version (training data MEDIUM); Step B audits `[features]` block to decide on `default-features = false`; Step C greps the resolved source for `EguiPlugin` shape (unit struct vs `EguiPlugin { enable_multipass_for_primary_context: bool }`), `EguiContexts::ctx_mut()` return type (`&mut Context` vs `Result<&mut Context, _>`), and quantifies expected Cargo.lock additions. **Why:** training data is MEDIUM for bevy_egui specifics; on-disk verification costs ~15 min and prevents an unbounded debugging session if the version is wrong. **How to apply:** any future plan adding a new bevy_xxx ecosystem crate (especially one with multiple Bevy-version branches) follows the same A/B/C gate. Halt + escalate to user if Step A's dry-run fails to resolve a Bevy 0.18-compatible version.

- **`MinimapPlugin` is a SIBLING of `DungeonPlugin` in `main.rs`, NOT nested inside `DungeonPlugin::build` and NOT in `src/plugins/ui/`.** Two `add_plugins` lines. **Why:** the map is data + view, not state-mutation; keeping it parallel avoids accidental coupling. Lowest-risk for Feature #23 save integration — `MinimapPlugin` owns `ExploredCells` and knows where to register save/load callbacks. `src/plugins/ui/` becomes a real aggregator only when Feature #19 Town UI lands. **How to apply:** new "data + view" subsystems (e.g., #25 settings UI minimap-related polish) consider sibling-plugin pattern first; only nest under DungeonPlugin if the subsystem is structurally inseparable from the dungeon (it isn't for the map).

- **`ExploredCells` is a `Resource` keyed by `(floor_no, x, y)`. Does NOT reset on `OnExit(GameState::Dungeon)`.** Cross-floor persistence is in the spec; F9 cycle preserves explored cells; reset is a Feature #23 / new-game concern only. **Why:** PlayerParty is despawned on `OnExit(Dungeon)` (`mod.rs:412-419`); a Component-on-PlayerParty model would lose the data. The `(floor, x, y)` key explicitly contradicts that lifecycle. **How to apply:** any future cross-floor persistent gameplay state (item-drop history, defeated encounter table, save-game checkpoints) follows the Resource pattern. Explicitly check the `OnExit(Dungeon)` pattern in `dungeon/mod.rs` before placing data on PlayerParty.

- **`handle_dungeon_input` becomes `pub(crate)` so `update_explored_on_move` can `.after(handle_dungeon_input)`.** Pitfall 3 — without explicit ordering, Messages buffer for the next frame and same-frame test assertions fail. **Why:** `MovedEvent` reader sees nothing until next frame unless explicit `.after` is set. The visibility change is the ONLY edit to `dungeon/mod.rs` (besides adding `pub mod minimap;`). **How to apply:** any future plugin that reads `MovedEvent` (Feature #13 cell triggers, Feature #16 encounters) follows the same `.after(handle_dungeon_input)` ordering. The `pub(crate)` exposure is now a stable contract — document the ordering coupling on `handle_dungeon_input`'s doc-comment so a future contributor doesn't make it private again.

- **All painters share a `MinimapSet` SystemSet ordered `.after(update_explored_on_move)`.** Pitfall 10 — without an ordering edge, Bevy's scheduler serializes the conflicting Resource access non-deterministically. **Why:** `update_explored_on_move` writes `ResMut<ExploredCells>`, both painters read `Res<ExploredCells>`. Without explicit ordering, sometimes painters see the latest update, sometimes not. **How to apply:** any future system pair (writer + reader on the same Resource within the same Update phase) declares a SystemSet for the readers and orders them after the writer.

- **`KnownByOther` variant declared in #10 with NO producer in v1.** Render with slight blue tint (`Color32::from_rgb(50, 50, 100)` vs `Visited`'s `Color32::from_rgb(60, 60, 70)`) so dev `show_full` is distinguishable from real exploration. Doc-comment names #12 (item-revealed map) / #20 (scry spell) as future producers. **Why:** zero scope creep, exhaustive match keeps future writers honest, no "unused variant" lint because the painter renders it. Forces #12/#20 to populate the existing variant rather than add it AND update every match site. **How to apply:** when planning #12/#20, the variant is already there — wire up the producer (an item-use system or spell-effect system) to insert `KnownByOther` for revealed cells. No painter changes needed.

- **Visited semantics for v1: destination cell ONLY (not adjacent line-of-sight).** Pitfall 4 — Option B (adjacent reveal) is a #25 polish requiring line-of-sight + door-aware traversal. **Why:** non-trivial algorithm; Option A satisfies the spec's "every MovedEvent" reading; visualization difference is small at floor_01 scale. **How to apply:** when #25 polish lands, the adjacent-cell reveal becomes a shared utility (line-of-sight checker + visible-cells query) used by both the minimap and any "peripheral vision" combat-detection feature.

- **`show_full` debug toggle is `#[cfg(feature = "dev")]`-gated symmetrically.** Field on `ExploredCells`, `toggle_show_full_map` system, painter's branch on `cfg!(feature = "dev") && explored.show_full` — all three are gated. Toggle key: F8 (F9 reserved for state cycler). **Why:** a runtime cheat-mode flag visible in shipping builds is a save-data injection / dark-zone-bypass concern. Symmetric `#[cfg(feature = "dev")]` gating is the established Druum pattern (`project/resources/20260501-102842-dev-feature-pattern.md`). **How to apply:** any future cheat / debug toggle (Feature #25 god-mode, Feature #20 instant-kill) follows symmetric gating: definition AND registration AND consumer-side branch all `#[cfg(feature = "dev")]`.

- **bevy_egui + dev `Camera2d` (debug HUD) coexistence is expected to work; manual smoke confirms.** Pitfall 9 — Druum's `--features dev` `spawn_debug_grid_hud` (mod.rs:781) spawns a `Camera2d { order: 1 }` for the dev HUD position text (top-LEFT). bevy_egui renders to its own pass and does NOT typically need an explicit Camera2d. Top-RIGHT minimap overlay placement was chosen specifically to avoid conflicting with this HUD. **How to apply:** any future overlay UI (Feature #14 HP bars, Feature #19 quest tracker) checks placement against the existing dev HUD position (top-left) and the minimap overlay (top-right). Bottom corners are still available.

- **Dark-zone gate is mandatory.** When `MovedEvent.to.x/y` lands on a `CellFeatures.dark_zone == true` cell, the subscriber skips the insert; the painter renders `?` regardless of state. Pitfall 8 — silent "no map update" alone is confusing UX; the `?` glyph is the visible feedback. **How to apply:** any future per-cell map decoration (Feature #25 colored markers, Feature #13 trap indicators) checks `dark_zone` first — dark-zone cells override every other rendering rule.

- **+1 Cargo dep cleanest-ship signal:** Cargo.toml diff is exactly +1 line for `bevy_egui`, Cargo.lock adds the egui tree (egui, epaint, ecolor, emath, ahash, possibly clipboard libs — Step C quantifies). NO unrelated transitive bumps (e.g., a `bevy_render` patch bump from upward unification means STOP and investigate). **Why:** this is the first non-trivial dep since Feature #5; extra care on the manifest diff is warranted. Past Δ deps = 0 features (#7, #8, #9) had a different "cleanest-ship signal" (zero Cargo.toml change). **How to apply:** any future +1-dep feature uses this same +1-line + bounded-lock-tree signal.

- **Test plan: Layer 1 (~3) + Layer 2 (~5) + Layer 2b (~3) ≈ 8 net new tests.** Layer 1 pure helpers (cell-rect math) — may be 0 if math stays inlined in `paint_floor_into`. Layer 2 (App + minimap, no leafwing) for the subscriber: uses `init_resource::<ButtonInput<KeyCode>>()` bypass, writes `MovedEvent`s directly to `Messages<MovedEvent>` via `.write(...)`. Layer 2b (App + full leafwing chain `MinimalPlugins + StatesPlugin + InputPlugin + ActionsPlugin + MinimapPlugin`) for the open/close handler. Manual smoke for the painter (egui requires render context). **How to apply:** baseline measurement (`grep -c '#\[test\]' src/**/*.rs`) is Step 0 — do NOT trust historical baselines; the actual count varies by feature.

- **Atomic commit boundaries (8 total):** (1) Cargo deps; (2) EguiPlugin in main.rs; (3) minimap.rs skeleton + `pub(crate)` change combined; (4-5 merged with 3); (4 actual) MovedEvent subscriber + Layer 2 tests; (5 actual) painter systems + helper; (6) open/close handler + Layer 2b tests; (7) `#[cfg(feature = "dev")]` show_full toggle; (8) docs + Implementation Discoveries. Each commit compiles; `cargo test` passes at every checkpoint. **How to apply:** future feature implementations follow the same per-logical-change commit boundary discipline (#7/#8/#9 precedent).
