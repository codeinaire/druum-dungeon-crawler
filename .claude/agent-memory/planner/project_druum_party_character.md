Feature #11 introduces the character data layer: 12 components + `PartyMember` marker + `PartyMemberBundle` + pure `derive_stats` + `PartySize` resource + `#[cfg(feature = "dev")]`-gated `spawn_default_debug_party` system + `core.classes.ron` with 3 classes (Fighter/Mage/Priest). Δ deps = 0; `Cargo.toml`, `Cargo.lock`, and `src/main.rs` all byte-unchanged. Several decisions here will shape downstream features:

- **Components-as-data with serde-from-day-one is non-negotiable.** Every one of the 12 components derives `Component, Reflect, Serialize, Deserialize, Default, Debug, Clone, PartialEq` (plus `Copy + Eq + Hash` where the inner type allows; `DerivedStats` and `Equipment` skip `Hash` for forward stability, `ActiveEffect` skips `Eq + Hash` because of the `f32 magnitude` field). **Why:** Feature #23 (save/load) is the trust-boundary consumer of these derives; bolting them on later means re-touching every save file. Per research §Pitfall 5. **How to apply:** any new component added in #12-#22 follows the same derive set. NO custom `Serialize` impls — let serde derive handle it so the attack surface is bounded.

- **`Equipment = Option<Handle<ItemAsset>>` per slot, NOT `Option<Entity>`.** Eight slots: `weapon, armor, shield, helm, gloves, boots, accessory_1, accessory_2`. **Why:** `Handle<T>` serializes as an asset-path string for free in #23 (no `MapEntities` retrofit). Per-instance state (enchantment, durability, custom name) lands as a separate `ItemInstance` entity in #12 — the alternative path explicitly designed at planning time. **How to apply:** Feature #12 (inventory) wraps `Handle<ItemAsset>` in an `ItemInstance` component on a child entity for per-instance state. `Equipment` slots stay `Handle<ItemAsset>` for the base reference. Save/load (#23) gets one less `MapEntities` impl to maintain.

- **`derive_stats(&BaseStats, &[ItemStatBlock], &StatusEffects, level: u32) -> DerivedStats` is PURE.** No `Mut<T>`, no `Query`, no `Res`, no entity lookups, no asset access, no randomness. Returns `current_hp = max_hp` and `current_mp = max_mp` as defaults; **callers clamp** per their semantics (level-up resets to max; equip-change is `min(old_current, new_max)`). **Why:** purity makes the function trivially testable; the caller-flatten pattern (`Equipment + Assets<ItemAsset> -> &[ItemStatBlock]`) keeps the asset access at the system layer where it belongs. Per OQ1. **How to apply:** Features #14 (level-up) and #15 (equip-change, status-tick) are the v1 callers; both write back `derived.max_hp = new.max_hp; derived.current_hp = derived.current_hp.min(new.max_hp);` (or full-heal on level-up). DO NOT trigger `derive_stats` recompute on `DerivedStats` change-detection — that's an infinite loop.

- **`derive_stats` uses `saturating_add` and `saturating_mul` everywhere.** Trust-boundary mitigation against malicious `classes.ron` / `items.ron` data (`hp_per_level: u32::MAX` overflow, etc.). **How to apply:** any future stat-aggregation function (#15 combat damage formula, #20 spell power) follows the same saturating-arithmetic discipline.

- **`Race` declares 5 variants (Human, Elf, Dwarf, Gnome, Hobbit), only Human used in v1. `Class` declares 8 variants (Fighter, Mage, Priest, Thief, Bishop, Samurai, Lord, Ninja), only first 3 have `ClassDef` entries.** **Why:** locks discriminant order so save-format stability is preserved when later features author the missing entries. Per Decisions 1 + 2. **How to apply:** when #19 (character creation) adds player-choice for races / classes, the enum variants are already there — just author the missing `ClassDef` entries in `core.classes.ron` and unlock the unused `Race` variants in the character creation UI. NO enum changes needed. `derive_stats` and `ClassTable::get` use wildcard arms or `Option` returns — never exhaustive `match` against `Class` without a wildcard.

- **`Dead` is a variant of `StatusEffectType`, NOT a separate marker component.** v1 set: Poison, Sleep, Paralysis, Stone, Dead. **Why:** keeps the status-effect surface to one component (`StatusEffects`); `derive_stats` zeros `max_hp`/`max_mp` when `Dead` is present; combat (#15) checks `status.has(StatusEffectType::Dead)`. Per Decision 7. **How to apply:** any future "permanent" character state (Petrified, Aged, Cursed) considers the `StatusEffectType` variant pattern first; only carve out a separate marker component if the state has zero stat impact AND zero turn-based decay (e.g., a "tutorial-only" debug marker).

- **No magnitude-modifying status effects in v1.** v1 `StatusEffectType` variants are all gates (Sleep/Paralysis/Stone) or tick-on-turn (Poison) or final (Dead). The `ActiveEffect.magnitude: f32` field is part of the schema for #15 buffs but is unread by v1 `derive_stats`. **Why:** v1 derive_stats is trivially order-independent (no magnitude branches), so the order-independence test from research line 583 is **deferred to #15**. **How to apply:** Feature #15 adds buff variants (AttackUp, DefenseUp, etc.) AND adds the `derive_stats_status_order_independent` test at the same time. Pick a composition rule (e.g., "sum magnitudes within an effect type, then apply once") and document on the `magnitude` field.

- **`PartySize::default() == 4` and is a HARD CAP.** `spawn_default_debug_party` refuses the (n+1)th spawn. Consumers query `Res<PartySize>` for capacity checks. **Why:** Wizardry/Etrian Odyssey/Grimrock standard; matches the 4-character debug party. Per Decision 6. **How to apply:** Feature #19 character creation MAY mutate the resource (e.g., to allow 1-4 player choice) — change the gate to "soft default" only if/when needed by #19's UX.

- **Debug-party gate = `#[cfg(feature = "dev")]` (NOT `cfg(debug_assertions)`); trigger = `OnEnter(GameState::Dungeon)` (NOT `OnEnter(Loading)`).** Roadmap line 596+629 says otherwise; we deviate from both. **Why:** project precedent (`cycle_game_state_on_f9`); avoids the asset-not-yet-loaded race in `OnEnter(Loading)` (research §Pitfall 3). The dev gate is symmetric (definition + registration both gated). Per Decision 5. **How to apply:** any future debug-only spawner (#15 enemy-roster preview, #20 spell library, etc.) follows the same `#[cfg(feature = "dev")]` + `OnEnter(...specific state...)` + idempotence-guard (`if !existing.is_empty() { return; }`) pattern. NEVER use `cfg(debug_assertions)` for game-content gating.

- **Idempotence guard on every `OnEnter(...)`-triggered spawn.** `spawn_default_debug_party` checks `if !existing.is_empty() { return; }` before spawning. F9 cycler can re-trigger any `OnEnter(...)`; without the guard, duplicates accumulate. **Why:** F9 dev cycler is a state cycler, not a "new game" trigger. **How to apply:** any future `OnEnter(GameState::*)` spawn-ish system follows the same guard pattern. Reset (when needed) is a Feature #23 / new-game concern only.

- **Single `src/plugins/party/character.rs` production file (Decision 4).** No `inventory.rs` / `progression.rs` despite roadmap line 609. **Why:** project precedent (#9, #10) is to NOT pre-split. **How to apply:** Feature #12 creates `inventory.rs` when the first inventory system lands; Feature #14 creates `progression.rs` when the first progression system lands. If `character.rs` exceeds 800 LOC during #11 implementation, extract a `tests.rs` sibling FIRST (per #9 precedent), then revisit `inventory.rs` / `progression.rs` only when their first system arrives.

- **`src/data/classes.rs` imports from `src/plugins/party/character.rs` — one-way reverse dep.** `ClassDef.id: Class` and `ClassDef.starting_stats: BaseStats` reference component types that live in `character.rs`. Project convention is `data/` is leaf; this is the first inversion. **Why:** research-recommended; the alternative (move `Class` and `BaseStats` to `data/`) was rejected as more disruptive (would split character types arbitrarily). NO cycle: `character.rs` does NOT import from `classes.rs`. **How to apply:** when later features add value-types-shared-between-plugin-and-data, follow the same one-way import pattern (data imports from plugin). If a cycle would form, restructure: move the shared type to `data/` and re-export.

- **`ItemAsset` and `ItemStatBlock` are stubbed in `src/data/items.rs` in #11 (alongside the existing `ItemDb`).** `ItemAsset { stats: ItemStatBlock }` is the v1 schema — empty otherwise. Feature #12 fleshes out the per-item state. `ItemStatBlock` has 8 `u32` fields (attack, defense, magic_attack, magic_defense, accuracy, evasion, hp_bonus, mp_bonus), all `#[serde(default)]`. **Why:** `Equipment` slots need `Handle<ItemAsset>` to resolve as a type at compile time; `derive_stats` needs `&[ItemStatBlock]` for tests. Per Pitfall 8. **How to apply:** Feature #12 adds `enchantment_level: u8`, `durability: u32`, `custom_name: Option<String>`, etc. to `ItemAsset` (or wraps the handle in an `ItemInstance` entity model). The existing `ItemStatBlock` field stays — #14 may add more stat fields.

- **`assets/classes/core.classes.ron` path is fixed by Feature #3's loader (`assets/classes/core.classes.ron`). Roadmap line `assets/data/classes.ron` is wrong.** `LoadingPlugin` is FROZEN post-#3 — no loader edits. **How to apply:** any future class-related asset (e.g., a `core.subclasses.ron` for prestige classes) reuses the existing loader registration in `loading/mod.rs:100` — no `LoadingPlugin` edits needed.

- **`bevy::utils::HashMap` is REMOVED in 0.18; use `std::collections::HashMap`.** #11 doesn't need a HashMap (the 8-class roster uses `Vec::iter().find()`), but the convention is locked. **How to apply:** any future class/item/spell lookup hot path (#14 progression with thousands of items, #20 spell library) uses `std::collections::HashMap` — never `bevy::utils::HashMap`.

- **`AssetEvent<T>` is a `Message` in 0.18, not an `Event`.** Not used by #11, but flag for #12/#19 implementers wiring balance hot-reload. **How to apply:** consume `AssetEvent` via `MessageReader<AssetEvent<T>>`, NOT `EventReader<AssetEvent<T>>`. Same trap as `MovedEvent` in #7 and `StateTransitionEvent` in #2.

- **`tests/class_table_loads.rs` mirrors `tests/dungeon_floor_loads.rs` for the loader-divergence regression test.** Asserts `table.classes.len() == 3` and that `get(Class::Fighter | Mage | Priest)` returns the authored shapes. Catches `ron 0.11` (loader) vs `ron 0.12` (unit-test) divergence per memory `project_druum_dungeon_grid` ("tests/ directory is the home for App-level integration tests"). **How to apply:** any future RON schema in `data/` (item DBs in #12, enemy DBs in #15, spells in #20) gets one App-level integration test in `tests/` mirroring this pattern.

- **Test count baseline (PRE-#11):** record at Step 0 of the plan via `grep -rc '#\[test\]' src/`. Expected delta after #11: ~+11 lib tests (8 in `character.rs::tests` + 2 in `classes.rs::tests` + 1 in `items.rs::tests`), +1 integration test (`tests/class_table_loads.rs`). Final verification: `BASELINE_LIB + 11 ± 1` lib + 4 integration. **How to apply:** the implementer must `grep` the actual count at Step 0 — do NOT trust historical baselines.

- **Δ deps = 0 cleanest-ship signal.** `Cargo.toml`, `Cargo.lock`, and `src/main.rs` MUST be byte-unchanged. PartyPlugin is already at `main.rs:32` (the `add_plugins` tuple) as an empty stub; the changes go INSIDE `PartyPlugin::build` in `mod.rs`, not in `main.rs`. Final `git diff Cargo.toml`, `git diff Cargo.lock`, and `git diff src/main.rs` MUST all show empty output. **How to apply:** any future feature that "wires up an existing empty plugin" follows this pattern — verify the plugin is already in `main.rs` BEFORE writing any `add_plugins` line (likely won't be needed).

- **Atomic commit boundaries (8 + 1 docs commit):** (1) baseline only — no code; (2) `items.rs` extended with `ItemAsset` + `ItemStatBlock` + round-trip test; (3) `character.rs` (12 components + Bundle + `derive_stats` + `PartySize` + 8 Layer 1 tests) + `classes.rs` (replaced with real schema + 2 tests) — combined commit because they're mutually dependent (the reverse-dep import); (4) `party/mod.rs` (PartyPlugin::build); (5) `data/mod.rs` (re-exports); (6) `core.classes.ron` (3-class data); (7) `tests/class_table_loads.rs` (integration test); (8) Implementation Discoveries docs. Each commit compiles; `cargo test` and `cargo test --features dev` pass at every checkpoint. **How to apply:** future feature implementations follow the same per-logical-change commit boundary discipline.
