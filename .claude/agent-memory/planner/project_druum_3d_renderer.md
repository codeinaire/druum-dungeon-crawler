Feature #8 replaces Feature #7's `TestSceneMarker` placeholder scene with real 3D dungeon geometry generated from `Res<Assets<DungeonFloor>>`. Several decisions here will shape downstream features:

- **Per-edge wall iteration rule: render `north + west` of every cell, plus `south` on bottom row (y == height-1) and `east` on right column (x == width-1).** **Why:** walls are stored twice in `WallMask` (cell A's east == cell B's west); naive per-cell rendering would double-render every interior wall (z-fighting + 2× geometry cost). North + west is the canonical convention; symmetric to south + east; pick one and stick. **How to apply:** Feature #13 (door visuals/animations) keys off the same iteration — when a door opens, find the entity that owns the wall plate at this cell's iteration position, NOT the paired side. Future authoring tools (#24 dungeon editor) should respect the convention or auto-translate.

- **`OneWay` walls: render only on the side stored as `Solid`. The `OneWay` side gets no geometry.** **Why:** `validate_wall_consistency` permits `OneWay`/`Solid` as the one allowed asymmetry. The blocking side needs visual feedback (player sees a wall when approaching from the wrong direction); the passable side needs nothing (player walks through it). Falls out for free from `wall_material(OneWay) = None`. **How to apply:** Feature #13's directional-arrow icon on the OneWay side's floor (telegraphing the asymmetry) is a separate floor-decal layer — it does NOT make the wall renderable. Feature #10 auto-map should render the OneWay arrow on the floor symbol, not as a wall.

- **`pub const CELL_HEIGHT = 3.0`, `WALL_THICKNESS = 0.05`, `FLOOR_THICKNESS = 0.05` in `src/plugins/dungeon/mod.rs`.** **Why:** 3.0 wall height matches `EYE_HEIGHT = 0.7` from #7's doc-comment, gives a 1.5× corridor-width-to-wall-height ratio (Wizardry-tight feel). 0.05 thickness avoids degenerate-volume Cuboids (0.0 produces undefined normals) and z-fighting with adjacent slabs while remaining invisible at typical viewing distances. **How to apply:** Feature #9 (atmosphere) tunes lighting against these constants; Feature #25 polish may bump thickness if z-fighting surfaces from new viewing angles. Do NOT redeclare in submodules — these are the source-of-truth constants alongside `CELL_SIZE` and `EYE_HEIGHT`.

- **Floor + ceiling per-tile (not merged slab). Walls per-edge (not per-cell).** **Why:** per-tile floor lets Feature #13 swap material per cell for spinner icons, dark-zone tints, anti-magic-zone tints without breaking geometry. Per-edge walls correctly handle `OneWay` and avoid double-rendering. Per-cell wall iteration would either lose `OneWay` asymmetry or require post-hoc deduplication. **How to apply:** Feature #13's per-cell visual variation logic spawns ADDITIONAL entities (decals on top of the floor tile), not new floor tiles — the per-tile spawn pattern means each cell's floor is already a single entity reachable by grid coords. Feature #9's per-cell torch placement attaches a `PointLight` as a child of the floor tile entity.

- **Lighting: scene-wide `DirectionalLight` (no shadows) + `GlobalAmbientLight` resource override on OnEnter, restored to `GlobalAmbientLight::default()` on OnExit.** **Why:** Etrian Odyssey-class atmosphere; player-attached torch is reserved for #9; `AmbientLight` in Bevy 0.18 is per-camera Component, `GlobalAmbientLight` is the resource (master research is wrong about this; corrected by #8 research). **How to apply:** Feature #9 lighting overhauls this with shadow-mapping, fog, and possibly a player-attached `PointLight`. Feature #18 Town has its own ambient; the OnExit→default convention is "every state owns its ambient on entry; OnExit resets to default." Future states should follow the same pattern; do NOT layer ambient overrides without OnExit resets.

- **`DungeonGeometry` marker component on every spawned mesh + light entity (top-level, no parent-child hierarchy).** **Why:** `Query<Entity, With<DungeonGeometry>>` despawn-on-OnExit is simpler than walking a parent-child tree, and `commands.entity(e).despawn()` is recursive in 0.18 anyway (handles future child-attachments). **How to apply:** Feature #9 atmosphere + #13 cell features add MORE `DungeonGeometry`-tagged entities for cleanup symmetry. Decals, torches, debug overlays — everything spawned during Dungeon state carries this marker, even if they have a feature-specific component too.

- **`wall_transform(grid_x, grid_y, dir) -> Transform` helper with `Quat::IDENTITY` rotation; orientation comes from mesh choice (NS vs EW), not transform.** **Why:** two natively-oriented Cuboid meshes (`(CELL_SIZE, CELL_HEIGHT, WALL_THICKNESS)` for NS, `(WALL_THICKNESS, CELL_HEIGHT, CELL_SIZE)` for EW) is simpler than rotating one mesh by `Quat::from_rotation_y(FRAC_PI_2)` and avoids subtle normal/lighting issues. Negligible cost (one extra mesh handle in the global handle cache). **How to apply:** Feature #13 wall-modification systems (e.g., door open/close animation) read the `Transform` from the wall plate entity directly; the transform's translation is on the cell edge, no rotation to compose.

- **`wall_material(WallType, &solid, &door, &locked) -> Option<Handle<StandardMaterial>>` is a free `fn`, not a closure.** **Why:** testable without an `App`. Feature #4's WallType variants map: `Open|OneWay → None`; `Solid|SecretWall|Illusory → solid handle` (player can't visually distinguish — reveal is #13); `Door → door handle`; `LockedDoor → locked handle`. **How to apply:** Feature #13's reveal mechanic mutates `WallType` (e.g., `SecretWall → Open` after discovery, `Illusory → Open` after walking through, `LockedDoor → Door` after key use); the next OnEnter(Dungeon) regenerates geometry with the new variant — no in-place mesh swap needed. If Feature #13 wants instant feedback (reveal mid-state), add a `WallTypeChanged` event and a system that despawns + respawns the affected wall plate.

- **v1 wall color palette is subjective and tunable via Step 13 visual smoke.** Solid grey `(0.50, 0.50, 0.55)`, Door brown `(0.45, 0.30, 0.15)`, LockedDoor dark red `(0.55, 0.20, 0.15)`, floor warm dark stone `(0.30, 0.28, 0.25)`, ceiling cool dark stone `(0.20, 0.20, 0.22)`. **Why:** placeholder colors until #9 textures land; the implementer iterates if visuals are unclear. **How to apply:** Feature #9 replaces flat colors with PBR textures (`base_color_texture`, `normal_map_texture`); the WallType-to-material mapping moves from per-color to per-texture but the function shape stays the same.

- **No module split: `src/plugins/dungeon/mod.rs` stays single-file at ~1175 LOC after Feature #8.** **Why:** consistent with Feature #7's single-file decision; gameplay logic + rendering doesn't naturally split because they share so much state (constants, marker components, `OnEnter/OnExit` cleanup). Splitting now would create churn that #9 (atmosphere) might want to undo. **How to apply:** if Feature #9 (atmosphere) genuinely needs a `render.rs` submodule (e.g., separate fog system, post-processing), that's the right time to split — extract `spawn_dungeon_geometry`, `wall_transform`, `wall_material`, the geometry constants, and `DungeonGeometry` together. Movement systems (`spawn_party_and_camera`, `handle_dungeon_input`, `animate_movement`) stay in `mod.rs`.

- **`floor_01` regression entity count: 121 = 36 floor + 36 ceiling + 48 walls + 1 directional light.** Walls breakdown: 14 north + 22 west + 6 south (bottom outer edge, all Solid) + 6 east (right outer edge, all Solid). **Why:** integration test in `tests/dungeon_geometry.rs` pins this count as a regression guard; future asset edits surface as test failures with documented math. **How to apply:** any planner editing `assets/dungeons/floor_01.dungeon.ron` (e.g., adding a new authoring example) MUST recompute this count and update the test assertion + comment. The test docstring derives the math from the canonical iteration rule.

- **Doc-comment fix at `src/data/dungeon.rs:18` is the SINGLE allowed exception to the data/dungeon.rs freeze.** Edit: `world_z = -grid_y * cell_size` → `world_z = +grid_y * cell_size`. **Why:** Feature #7's review (LOW finding) explicitly deferred to #8. **How to apply:** future planners do NOT modify `data/dungeon.rs` further unless explicitly delegated by a review or new architectural directive. The world-coordinate convention is now stable across `data/dungeon.rs` and `plugins/dungeon/mod.rs`.
