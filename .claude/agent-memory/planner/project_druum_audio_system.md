Feature #6 introduces `AudioPlugin` at `src/plugins/audio/mod.rs` plus submodules `bgm.rs` and `sfx.rs`, an `AudioAssets` collection in `LoadingPlugin`, and 10 placeholder silent .ogg files under `assets/audio/{bgm,sfx}/`. Several decisions here will shape downstream features:

- **Roadmap deviation: built-in `bevy_audio`, NOT `bevy_kira_audio`.** Roadmap line 358 pre-committed to `bevy_kira_audio = "0.25"`. Research (HIGH-confidence on-disk verification of `bevy_audio-0.18.1/`) deviated: Bevy 0.18 ships `Volume::fade_towards` + an official `examples/audio/soundtrack.rs` crossfade pattern, and the `"3d"` feature umbrella in `Cargo.toml` already pulls in `bevy_audio + vorbis` transitively. **Why:** zero new deps, zero version-axis maintenance, zero kira-vs-built-in coexistence pitfall (the `"3d"` umbrella would force significant Cargo.toml surgery to disable built-in audio). Same shape as Feature #3's `moonshine-save` deviation. **How to apply:** if a future feature wants kira-only capabilities (HRTF spatial audio, sample-accurate music quantization), the marker-component design lets us swap the `Sfx` channel's playback engine without touching gameplay callers â€” defer to Feature #25 audio-polish, do not pre-emptively add kira.

- **Channels are zero-sized marker `Component`s, NOT typed `AudioChannel<T>` resources.** `Bgm`, `Sfx`, `Ui`, `Ambient` derive `Component` only. Spawn pattern: `commands.spawn((AudioPlayer::new(handle), PlaybackSettings, Bgm))`. Query pattern: `Query<&AudioSink, With<Bgm>>`. **Why:** built-in `bevy_audio` has no concept of channels; the kira-style typed-channel resource is not native. **How to apply:** every audio spawn site in later features (#7 footsteps, #15 attack hits, #9 ambient) MUST attach the appropriate marker. Future contributors should NOT introduce a `Res<AudioChannel<Bgm>>` style API to "match kira" â€” the marker pattern is intentional. The `mod.rs` module-level doc documents this convention.

- **`SfxRequest` is a `Message`, not an `Event`** (Bevy 0.18 family rename, same trap that bit `StateTransitionEvent`/`AssetEvent`/`KeyboardInput`). Read with `MessageReader<SfxRequest>`. Register with `app.add_message::<SfxRequest>()`. **How to apply:** Feature #7 footstep code writes `MessageWriter<SfxRequest>::write(SfxRequest { kind: SfxKind::Footstep })` directly â€” no helper function. Feature #15 attack-hit code does the same. Direct `MessageWriter` is one line and clear. Adding a `play_sfx(commands, handle, kind)` helper was rejected (research Â§RQ8) because it couples gameplay code to audio internals.

- **`SfxKind` is a closed enum** with five variants: `Footstep`, `Door`, `EncounterSting`, `MenuClick`, `AttackHit`. Adding a new SFX kind requires (a) adding the variant, (b) adding the corresponding `Handle<AudioSource>` field to `AudioAssets`, (c) adding the `match` arm in `handle_sfx_requests`. The compiler enforces all three (exhaustive match). **How to apply:** when later features need new SFX, extend the enum â€” do NOT pass arbitrary `Handle<AudioSource>` through `SfxRequest`. The closed enum is the trust boundary.

- **`AudioAssets` is a NEW collection, NOT bundled into `DungeonAssets`.** Loaded by `bevy_asset_loader` alongside `DungeonAssets` via `LoadingState::load_collection::<AudioAssets>()`. **Why:** missing .ogg should not block dungeon-data tests; per-feature isolation. **How to apply:** future asset families that have a distinct failure-mode footprint (e.g., font assets in #25, save-slot icons in #23) get their own `AssetCollection`. Do NOT bundle multi-domain assets into a single mega-collection.

- **Stateâ†’BGM mapping is hardcoded `match` in `play_bgm_for_state`.** Five lines, exhaustive over `GameState`. RON-driven `Resource<HashMap<GameState, Handle<AudioSource>>>` is deferred to Feature #25 polish (research Â§RQ7 resolved). **How to apply:** when adding a new `GameState` variant in a later feature, the compile error in `play_bgm_for_state`'s match surfaces the choice "what BGM plays here?" â€” pick one of the existing `bgm_*` handles or add a new field to `AudioAssets`. Do NOT make `play_bgm_for_state` non-exhaustive with a `_ => {}` fallback.

- **Crossfade duration is `pub const FADE_SECS: f32 = 1.0;` in `bgm.rs`.** Symmetric fade-out + fade-in. **Why:** task brief specified ~1 sec. `Volume::fade_towards` does linear interpolation in linear-volume domain (matches the official example). **How to apply:** Feature #25 audio-polish may add easing curves or per-state fade durations â€” that's the right time to revisit. v1 ships one constant.

- **`PlaybackSettings::DESPAWN` for one-shot SFX, `PlaybackMode::Loop` for BGM with `FadeOut` despawn for termination.** Verified at `bevy_audio-0.18.1/src/audio.rs:105-109`. **Why:** `DESPAWN` is the canonical play-once idiom (no manual cleanup needed). `Loop` + `FadeOut` despawn is the canonical termination guarantee for fading BGM (research Â§Pitfall 6 â€” without despawn, a fading loop runs forever at silent volume). **How to apply:** any future audio spawn that's a one-shot uses `PlaybackSettings::DESPAWN`. Any future looping audio that needs to terminate uses `FadeOut` then despawn (or its own equivalent termination component). Never use `PlaybackSettings::ONCE` without an explicit despawn system â€” that leaks entities.

- **`audio_output_available` run condition is false on headless CI.** Tests assert MARKER components (`Bgm`, `Sfx`, `AudioPlayer`) NOT `AudioSink` (which is added by `bevy_audio`'s `PostUpdate` systems gated by audio output, skipped on CI). **How to apply:** Feature #7+ tests that spawn audio entities follow the same pattern â€” assert markers via `Query<Entity, With<Bgm>>::iter().count()`, never `Query<&AudioSink>::single()`. Production code reading `AudioSink` must be tolerant: `if let Ok(sink) = q.single()` not `q.single().unwrap()`.

- **One-frame state transition deferral applies to BGM tests.** `next.set(GameState::Town)` queues; `play_bgm_for_state` (gated by `state_changed::<GameState>` in `Update`) observes the new state on the next frame's `Update`. Tests need `app.update()` Ă— 2 after `next.set(...)`. Same pattern as the F9 test in `src/plugins/state/mod.rs:130-131`. **How to apply:** when later features write tests that drive `next.set(...)` and then assert on systems that read `state_changed::<GameState>` or `in_state(...)`, always update at least twice.

- **No `dev`-only audio hotkeys in v1.** No mute toggle, no per-channel volume override, no dev-cycler for BGM. **How to apply:** if a later feature wants debug audio hooks (e.g., `cycle_audio_volume_on_F8`), MUST symmetric-cfg-gate per `project/resources/20260501-102842-dev-feature-pattern.md` (both function definition and `add_systems` call). The discipline applies even though Feature #6 ships nothing cfg-gated.

- **Asset format is `.ogg` (Vorbis, 1-second silent placeholders, ~1-3 KB each).** Generated via `ffmpeg -f lavfi -i anullsrc=r=44100:cl=stereo -t 1 -c:a libvorbis -q:a 0` or `sox -n -r 44100 -c 2 silent.ogg trim 0 1`. **Why:** Druum's `Cargo.lock:4394-4401` already resolves `rodio + lewton` (the .ogg/Vorbis decoder); empty .ogg files panic in `rodio::Decoder::new`; silent files are licensing-risk-free. **How to apply:** Feature #25 audio-polish replaces silent files with real audible content via `git mv` + content swap. Do NOT add `mp3`/`flac`/`wav` Bevy features unless a real-content decision in #25 motivates them â€” `vorbis` (already on via `"3d"` umbrella) is sufficient.

- **`ChannelVolumes` resource is initialized to all-1.0 linear, not yet applied per-channel in v1.** A `apply_channel_volumes` system gated by `resource_changed::<ChannelVolumes>` is deferred to Feature #25 settings UI (research Â§Pitfall 7 acknowledges this v1 simplification). **How to apply:** Feature #25 adds (a) a settings UI volume slider, (b) the `apply_channel_volumes` system, (c) optionally persistence across app sessions. The Resource exists in v1 to lock in the API shape; the behaviour lands in #25.
