Feature #4 introduces the razor-wall grid data model in `src/data/dungeon.rs` (extends a Feature #3 stub) and establishes conventions that downstream features must follow:

- **Walls are stored TWICE per shared edge (`WallMask` per cell, all four sides).** **Why:** Hand-authoring 20 floor files before Feature #24's editor lands is the dominating cost; store-twice means every cell is self-contained (no neighbor lookups). The trade-off is one consistency-checking method (`DungeonFloor::validate_wall_consistency`) and one unit test per authored floor file. **How to apply:** when adding a new authored floor file in any later feature, add a unit test that calls `validate_wall_consistency()` on that file and asserts `Ok(())`. Never read the wall on a shared edge from one cell only — always go through `wall_between(a, b)` or `can_move(x, y, dir)` so the convention stays encapsulated.

- **Grid coordinates use y-DOWN screen convention. `Direction::North = (0, -1)`.** **Why:** RON files lay out `walls: [[row 0], [row 1], ...]` with row 0 at the top of the file = top of screen = "north"; auto-map (#10) renders the same way; classic Wizardry/Etrian floor maps put north at the top of the page. **How to apply:** Feature #7 (movement) computes `let new = (pos.0 as i32 + dir.offset().0, pos.1 as i32 + dir.offset().1)` and the offsets carry the y-down convention. Feature #8 (3D renderer) is the SINGLE place that converts grid-y-down to Bevy world-y-up, typically via `Vec3::new(x as f32 * cs, 0.0, -(y as f32) * cs)`. Do NOT introduce a "world-space Direction" — keep one `Direction`, one convention, with the conversion contained in #8.

- **`Direction` rotation methods are named `turn_left`, `turn_right`, `reverse`** (NOT `rotate_left`, `rotate_right`, `rotate_180`). **Why:** Matches §Pattern 2 of the source-of-truth research (`research/20260326-01-bevy-first-person-dungeon-crawler-rpg.md:395-421`) and Wizardry/Etrian vocabulary; consistent with first-person navigation language. **How to apply:** any future feature touching `Direction` uses `turn_left`/`turn_right`/`reverse`. Do not add `rotate_*` aliases.

- **`CellFeatures` is a struct-of-optionals with `#[serde(default)]`, NOT an enum.** Fields: `trap`, `teleporter`, `spinner`, `dark_zone`, `anti_magic_zone`, `encounter_rate`, `event_id`. **Why:** Cells frequently combine features (dark zone with a trap; anti-magic teleporter); enum-of-features cannot represent combos cleanly. `#[serde(default)]` lets empty cells emit as `()` in RON instead of enumerating every field. **How to apply:** when extending `CellFeatures` with new feature fields in later features, add another `Option<...>` or `bool` field with a `#[serde(default)]`-compatible default — never replace the struct with an enum.

- **Doors live in `WallType` (`Door`, `LockedDoor`), NOT in `CellFeatures`.** **Why:** doors are wall-side properties (between two cells), not cell-center properties; a cell can have doors on multiple sides. Modeling doors as a `CellFeatures` variant would either force a `Vec<DoorOnSide>` (duplicating `WallType::Door` already in `WallMask`) or break the `can_move` lookup model. **How to apply:** Feature #13 (door interactions) mutates `WallType` (e.g. `LockedDoor` -> `Door`); it does NOT touch `CellFeatures`.

- **`can_move` returns true on `Open | Door | Illusory | OneWay`, false on `Solid | LockedDoor | SecretWall` and out-of-bounds. Discovery state is NOT a parameter.** **Why:** `DungeonFloor` is a static, hot-reloadable, file-backed asset; mutable per-save discovery state lives in a separate resource (`Res<DiscoveredCells>`, owned by Feature #13). Mixing the two would couple the asset to gameplay state, break hot-reload, and bloat saves. **How to apply:** Feature #7 (movement) calls `can_move` for the asset-level check, then layers a `can_move_with_discovery(x, y, dir, &discovered)` helper for `SecretWall`. Feature #12 (inventory) layers a key-availability check for `LockedDoor`. The base `can_move` stays pure.

- **Spinners are `CellFeatures::spinner: bool` only.** No `SpinnerStyle` enum, no `telegraphed: bool`. **Why:** Roadmap §Resolved #4 says all spinners across all 20 floors are modern telegraphed (Etrian-style). A flag for a non-existent classic-mode is YAGNI; if Iron Mode adds classic spinners later, that becomes a global setting (`Res<IronModeSettings>`), not a per-cell field. **How to apply:** auto-map (#10), movement (#7), and combat (#15) all key off the `bool`. Do not propose a per-cell spinner config in any later feature.

- **`tests/` directory at the crate root is the home for App-level integration tests.** Feature #4 is the first feature to add one (`tests/dungeon_floor_loads.rs`), specifically to verify the `bevy_common_assets::RonAssetPlugin` (ron 0.11) loader path matches the unit-level `ron 0.12` round-trip. **Why:** the prior code reviewer (`feedback_ron_version_split.md`) flagged that unit tests exercise the `ron 0.12` path while production loads use `ron 0.11`; an App-level integration test is the only thing that catches loader-vs-stdlib divergence. **How to apply:** when later features add typed RON schemas with non-trivial bodies (item DBs in #11, enemy DBs in #11/#15, classes in #19, spells in #20), each gets one App-level integration test in `tests/` that loads its asset via `RonAssetPlugin` and asserts shape. The `bevy_asset_loader-0.26.0/tests/multiple_asset_collections.rs` pattern (with `MessageWriter<AppExit>` per the Bevy 0.18 Event/Message rename) is the canonical template.

- **`#[serde(default)]` at the STRUCT level on `CellFeatures` is non-negotiable.** Forgetting it forces every cell in the RON file to enumerate every field, which balloons file size and breaks the lazy-author intent. **How to apply:** any future struct-of-optionals asset schema (item modifiers, enemy resistances, spell components) that wants empty/partial entries to be authorable as `()` or partial constructors needs the same struct-level attribute.
