Feature #5 introduces `ActionsPlugin` at `src/plugins/input/mod.rs` and pins `leafwing-input-manager` (version resolved at implementation time via Step A gate). Several decisions here will shape downstream features:

- **Plugin name is `ActionsPlugin`, NOT `InputPlugin`.** **Why:** `bevy::input::InputPlugin` is reachable via `bevy::prelude::*`; naming our plugin `InputPlugin` forces per-file aliasing. **How to apply:** any future "the input plugin" reference is `ActionsPlugin`. Same naming-clash discipline as the `StatePlugin` vs `bevy_state::StatesPlugin` distinction in Feature #2.

- **`InputManagerPlugin::<T>::default()` is registered UNCONDITIONALLY in `ActionsPlugin::build`** â€” NO `.run_if(in_state(...))` on the plugin call. Plugins are not run-conditioned in Bevy 0.18 (verified at `bevy_app-0.18.1/src/plugin.rs` â€” no `run_if` method on `Plugin` trait). **Why:** the roadmap line 314's wording (`app.add_plugins(InputManagerPlugin::<DungeonAction>::default().run_if(in_state(GameState::Dungeon)))`) does not compile. **How to apply:** state-scoping is a *consumer-side* concern. Feature #7's `handle_dungeon_movement` system gets `.run_if(in_state(GameState::Dungeon))`; Feature #15's combat input systems get `.run_if(in_state(GameState::Combat))`. The `ActionState<T>` resources are always-present and always-updated; only the consumer systems gate themselves.

- **Per-context action enums (3 in v1: `MenuAction`, `DungeonAction`, `CombatAction`).** No mega-enum, no `TownAction` (Town reuses `MenuAction` for v1), no `DevAction` (deferred to first leafwing-routed dev hotkey). **Why:** type-system enforces state-scoping â€” a system reading `Res<ActionState<DungeonAction>>` in combat code is a compile error. **How to apply:** when a future feature adds a new game-state context with non-overlapping action vocabulary, add a new enum (`TownAction` when Town gets distinct movement in Feature #19+; `DevAction` when first leafwing-routed dev hotkey lands beyond F9). Same context can have multiple enums active because state-scoping is type-level.

- **`InputMap<T>` is a `Resource`, not a `Component`.** **Why:** Druum is single-player; the per-entity `Component` shape (leafwing 0.15+ idiom for couch co-op) adds a query layer for no current benefit. **How to apply:** consumers use `Res<ActionState<T>>`, not `Query<&ActionState<T>>`. Reassess at Feature #25 if multi-player ever scoped (would also affect rebinding UI design).

- **F9 cycler stays on `Res<ButtonInput<KeyCode>>` â€” DOES NOT route through leafwing.** Documented in `ActionsPlugin`'s module-level doc comment. **Why:** F9 is dev-only and never user-rebindable, so leafwing's main feature (rebinding) is unused; refactoring would require six cfg-gating points; the existing F9 test uses the `init_resource::<ButtonInput<KeyCode>>` bypass pattern which would need to switch to full `InputPlugin` + message injection. **How to apply:** future contributors should NOT try to "fix the inconsistency". The carve-out is intentional and explained in the doc comment. The F9 test pattern in `src/plugins/state/mod.rs:117-137` stays unchanged.

- **Tests for `ActionState<T>` use the OPPOSITE pattern from F9 test:** full `InputPlugin` + `Messages<KeyboardInput>` injection, not `init_resource::<ButtonInput<KeyCode>>` bypass. **Why:** leafwing's update reads `ButtonInput<KeyCode>` AFTER `keyboard_input_system` populates it from `KeyboardInput` messages. Without `InputPlugin`, leafwing has no fresh input to consume. Without message injection, `keyboard_input_system` clears the press at frame start before leafwing reads. **How to apply:** any future `ActionState<T>` test follows the Â§Step 8 template in the Feature #5 plan: `MinimalPlugins + StatesPlugin + InputPlugin + ActionsPlugin`, write `KeyboardInput` to `Messages<KeyboardInput>` via `.write(...)`, call `app.update()`, assert `ActionState::just_pressed(&action)`. The `KeyboardInput` 6-field layout is verified at `bevy_input-0.18.1/src/keyboard.rs:109-139`.

- **Default keymap conventions (research Â§RQ9 + task-spec resolved):** WASD + arrows for menu nav; WASD + arrows STRAFE in dungeon (modern Wizardry/Etrian); Q/E for turning (no arrow alternates); F for Interact (avoids TurnRight=E conflict); M for map; Tab for inventory; Escape for cancel/pause. **Why:** modern dungeon-crawler convention; arrows-strafe matches WASD primary; F avoids the E conflict cleanly. **How to apply:** Feature #25's settings UI rebinding stores user overrides on top of these defaults. Don't bind F1-F12 for gameplay (reserved for dev hotkeys); don't bind Backquote (reserved for future debug console).

- **`Actionlike` derive bound list (defensive superset):** `#[derive(Actionlike, PartialEq, Eq, Hash, Clone, Copy, Debug, Reflect)]`. **Why:** matches leafwing's documented bound family across versions 0.13-0.18; `Reflect` is free for unit enums per `reference_bevy_reflect_018_derive.md` and may be required for type registration depending on resolved version. **How to apply:** every action enum gets the full bound list. NO `Default` derive (action enums have no "default action").

- **Feature #5 escalation playbook:** if `cargo add leafwing-input-manager --dry-run` resolves a version that requires `bevy = "0.17"` or `bevy = "0.19"+`, HALT and escalate. Same pattern as Feature #3's `moonshine-save` swap. Do NOT downgrade Bevy. Options to surface: (a) wait for upstream, (b) community fork, (c) hand-roll (~150-300 LOC for `Actionlike` + `InputMap` + `ActionState`).
