Feature #9 adds atmospheric lighting (per-floor `DistanceFog`, per-cell torches with deterministic flicker, per-floor ambient brightness) on top of Feature #8's geometry. Several decisions here will shape downstream features:

- **`ColorRgb(f32, f32, f32)` wrapper struct, NOT `bevy::Color`, for any RON-deserialized color.** **Why:** `bevy::Color: Serialize/Deserialize` is gated behind `bevy_color/serialize` (verified `bevy_color-0.18.1/Cargo.toml:58-61`); Druum's `bevy = { features = ["3d", ...] }` does NOT enable it. Adding the feature would cascade 12 transitive features and modify Cargo.toml â€” violates the byte-unchanged guarantee. The wrapper is ~10 LOC and dodges the gate at zero feature-flag cost. `into_color()` clamps each channel to `[0.0, 1.0]` for trust-boundary safety. **How to apply:** any future RON schema with a color field (item glow, spell color, enemy aura) uses `ColorRgb`, not `Color`. Document the gate on the wrapper's doc-comment so a future contributor doesn't undo the workaround.

- **Carried torch is unconditional and untouchable. Cell torches stack additively under sconces; user explicitly accepts bright spots.** Properties are pinned: `intensity: 60_000.0`, `range: 12.0`, `color: srgb(1.0, 0.85, 0.55)`, `shadows_enabled: false`, parented to `DungeonCamera` via nested `children![]`. **Why:** user quote "the code is the source of truth so don't change it." Where the spec/research and the existing code disagree on the carried torch, trust the code. **How to apply:** Feature #25 polish may revisit the additive-stacking aesthetic, but only with explicit user re-approval. Future per-floor mood toggles (e.g., a `carried_torch: bool` in `LightingConfig`) were considered (Option C) and explicitly rejected for #9. Do not add the toggle without re-asking.

- **Carried torch flickers via `Torch { phase_offset: Ď€ }` â€” same `flicker_torches` system as cell torches.** **Why:** unifying the flicker pipeline avoids a special-case carrier system; `phase_offset = Ď€` desyncs the carried torch from the (0,0) cell torch (half a wavelength out of phase). **How to apply:** any future "additional carried light" (Feature #20 spell light, Feature #25 lantern variant) follows the same pattern: spawn with `Torch { base_intensity, phase_offset: <constant unique to this light> }` and let `flicker_torches` handle it.

- **`#[cfg(test)] mod tests` extracted to a sibling `src/plugins/dungeon/tests.rs` file. Production code stays single-file in `mod.rs`.** **Why:** user-proposed Option D, off-menu from research's A/B/C. Test code was the bulk of `mod.rs` size (~664 LOC); extracting it cuts the production file roughly in half without restructuring production code. Standard Rust pattern: `#[cfg(test)] mod tests;` declaration + sibling file body with `use super::*;`. **How to apply:** if Feature #25 polish or any later feature needs to extract production code (real submodules), it can do so without conflicting with this test split. Future features can ALSO follow this pattern â€” add tests inline initially, extract to a sibling `tests.rs` when the file gets unwieldy. Watch for `super::` imports in tests â€” they keep working because `tests` remains a child module.

- **`light_positions: Vec<TorchData>` and `lighting: LightingConfig` are the two new `DungeonFloor` fields. `data/dungeon.rs` exception #2 (and last).** **Why:** roadmap-mandated schema extension; `data/dungeon.rs` is otherwise frozen since Feature #4 (the doc-comment fix in #8 was the first exception). Both fields have `#[serde(default)]` so existing assets without these fields continue to load. **How to apply:** future schema extensions to `DungeonFloor` (e.g., music track per floor, Feature #6 deferred) require fresh research/planning + user directive. Do NOT extend in passing during a different feature.

- **`LightingConfig::default()` and `FogConfig::default()` are explicit `impl Default` blocks (NOT `#[derive(Default)]`).** Defaults: `LightingConfig { fog: FogConfig::default(), ambient_brightness: 1.0 }`; `FogConfig { color: ColorRgb(0.10, 0.09, 0.08), density: 0.12 }`. **Why:** `#[derive(Default)]` would set `ambient_brightness: 0.0` (pure black) and `density: 0.0` (no fog) â€” both wrong defaults. Hand-written impls return atmospheric starting values so floors omitting `lighting:` get sensible behavior. **How to apply:** any future config struct with semantically-meaningful non-zero defaults uses `impl Default` explicitly. `ColorRgb` and `TorchData` keep `#[derive(Default)]` (zeros are sane sentinels there).

- **Per-floor fog is added in `spawn_party_and_camera`, not a separate system.** `DistanceFog` goes on the inner `Camera3d` child of `PlayerParty`. Single floor read; the system already has `floors.get(...)` in scope. **Why:** avoids a second system reading the same floor handle; matches how `entry_point` is already consumed there. **How to apply:** Feature #10 auto-map's potential second Camera3d (minimap preview) needs its own fog (or `..default()` to skip) â€” fog is per-camera (`#[extract_component_filter(With<Camera>)]` at `bevy_pbr-0.18.1/src/fog.rs`). Don't assume the dungeon's fog applies to other cameras.

- **`DistanceFog` falloff is ALWAYS `Exponential { density }` â€” never default `Linear`.** `DistanceFog::default().falloff` is `Linear { 0.0, 100.0 }`, invisible at dungeon scale (12 world units across). Tests assert `matches!(fog.falloff, FogFalloff::Exponential { .. })` to catch this. **How to apply:** any future fog setup (Town floors, alternate dimensions) specifies `Exponential` explicitly. The tests pin the convention.

- **Per-cell torches spawn inside `spawn_dungeon_geometry`. Tagged BOTH `Torch` (flicker filter) AND `DungeonGeometry` (cleanup).** **Why:** sharing the spawn system means torches and walls iterate the same floor handle in one read; the existing OnExit `Query<Entity, With<DungeonGeometry>>` despawns torches automatically â€” no parallel cleanup. **How to apply:** any future per-cell entity (Feature #13 cell decorations, Feature #25 props) follows the same dual-tag pattern when it needs both gameplay-feature filtering and lifecycle cleanup.

- **Flicker formula is `1.0 + 0.10*sin(t*6.4 + phase) + 0.05*sin(t*23.0 + phase*1.7)`, clamped `[0.80, 1.20]`.** Two sines at incommensurate frequencies + per-entity phase offset = visually convincing flicker without a noise crate. Per-entity phase from `(x*31)^(y*17)` hash for cell torches; `f32::consts::PI` for the carried torch. **Why:** Î” deps = 0 (no `noise` crate); deterministic across playthroughs (no `rand`, no `Instant::now()`). **How to apply:** if a future feature needs different flicker character (faster/slower/more violent), tune the `0.10`/`0.05` coefficients or the `6.4`/`23.0` frequencies, but don't remove the clamp â€” Pitfall Â§Pitfall 5 (real torches vary 5-15%; >20% reads as "broken light bulb").

- **Shadow cap is authored, not coded.** Bevy 0.18 stable-sorts `shadows_enabled: true` lights by entity ID and truncates at `max_texture_array_layers / 6` (typically ~42). Spec's "4 per visible region" is achieved by setting `shadows: true` on at most 3-4 entries per asset â€” zero LOC, deterministic. **How to apply:** floor authors set `shadows: true` sparingly (3-4 per floor max). The `TorchData::shadows` doc-comment documents this. If "nearest 4 by distance" becomes important (Feature #25 polish), add an explicit ~30 LOC system; don't pre-emptively build it.

- **`tests/dungeon_geometry.rs` count: 124 = 36 floor + 36 ceiling + 48 walls + 4 torches.** Adding/removing torches in `floor_01.dungeon.ron` requires updating this count and its derivation comment. **Why:** integration test pins this as a regression guard; the assertion message derives the math so a failure is immediately understandable. **How to apply:** any planner editing `floor_01.light_positions` MUST recompute and update `tests/dungeon_geometry.rs:152`.

- **Carried torch is NOT tagged `DungeonGeometry`.** It's a grandchild of `PlayerParty` and despawns recursively when `PlayerParty` despawns (verified `bevy_ecs-0.18.1/src/system/commands/entity_command.rs:242-249`). **How to apply:** when counting `DungeonGeometry` entities, the carried torch never appears in the count. When counting `Torch` entities (cell + carried), it always does.

- **Test count baseline: 62 lib + 3 integration BEFORE Feature #9; 69 lib + 3 integration AFTER (assuming 4 new dungeon-module tests + 3 new data/dungeon tests + extended on-exit test).** Pipeline state's "61 lib" was off by one â€” `grep -c '#\[test\]' src/**/*.rs` gives 62. **How to apply:** future features should `grep` the actual `#[test]` count at planning time, not trust historical baselines. Update the baseline in plans accordingly.
