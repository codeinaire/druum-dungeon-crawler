Feature #7 fills the empty `DungeonPlugin` stub at `src/plugins/dungeon/mod.rs` with first-person grid movement, a child `Camera3d`, tween animation, `MovedEvent` Message, and placeholder test scene scaffolding. Several decisions here will shape downstream features:

- **World coordinate convention: `world_x = grid_x * CELL_SIZE`, `world_z = +grid_y * CELL_SIZE`, `world_y = 0` for ground.** **Why:** with `Direction::North = (0, -1)` from Feature #4, North movement decrements grid_y → -Z motion in world space, which matches Bevy's default camera-looking-direction (-Z). A camera "facing North" can use `Quat::IDENTITY`; turning right (to East) is `Quat::from_rotation_y(-FRAC_PI_2)`. **How to apply:** Feature #8's renderer reads from `dungeon/mod.rs::grid_to_world`. The stale `data/dungeon.rs:18` doc-comment proposing `world_z = -grid_y * cell_size` is forward-looking conjecture from Feature #4 that #7 overrides; do NOT modify that comment (avoid touching frozen-ish files unnecessarily). The `dungeon/mod.rs` source-of-truth supersedes.

- **`CELL_SIZE = 2.0`, `EYE_HEIGHT = 0.7`, `MOVE_DURATION_SECS = 0.18`, `TURN_DURATION_SECS = 0.15` as `pub const` in `src/plugins/dungeon/mod.rs`.** **Why:** 2.0 is genre-correct corridor scale (Wizardry/Etrian feel — wider 4.0 produces "ballroom" feel at human scale); 0.7 produces "facing-forward" feel relative to Feature #8's planned 3.0 wall height; 0.18s/0.15s tween durations are tunable in Feature #25 polish. **How to apply:** Feature #8's wall geometry uses `CELL_SIZE` from the same file; do NOT redeclare. If a `dungeon/spatial.rs` ever splits these out, migrate all four constants together.

- **Logical state updates IMMEDIATELY at input-commit time; visual state lerps via `MovementAnimation`.** `GridPosition` and `Facing` mutate on the same frame as `actions.just_pressed(...)` after `can_move` clears. The `MovementAnimation` component then animates Transform from the old world_pos to the new one over 0.18s (or 0.15s for turn-only). **Why:** master research §Anti-Pattern (line 982) calls for separating logical and visual state; downstream consumers (#13 cell triggers, #16 encounter rolls) react to the new GridPosition immediately, not 180ms later. **How to apply:** Feature #13's cell-feature trigger system listens to `MessageReader<MovedEvent>` and reads `GridPosition` directly from the player entity — both are already at the destination by the time the message fires. Feature #16's encounter-roll system does the same.

- **`MovedEvent` is a `Message`, NOT an `Event`** (Bevy 0.18 family-rename, same trap as `SfxRequest`/`KeyboardInput`/`StateTransitionEvent`). Fields `{ from: GridPosition, to: GridPosition, facing: Direction }`. Registered with `app.add_message::<MovedEvent>()`. **How to apply:** downstream features reading move events use `MessageReader<MovedEvent>`. Adding additional fields (e.g., `was_blocked: bool` for wall-bumps, `kind: MoveKind` for translation-vs-turn) is a future-feature breaking change — the v1 contract is "translation moves only, fields are stable."

- **No input buffering — drop input during animation via `Without<MovementAnimation>` query filter.** `handle_dungeon_input` queries `Query<..., (With<PlayerParty>, Without<MovementAnimation>)>`. A second key press during a tween is dropped at the query level; no `PendingMove` resource. **Why:** the simplest implementation; matches Etrian Odyssey / Wizardry Trilogy Remaster / Legend of Grimrock. Migration to a buffer in v2 is local to one query filter swap. **How to apply:** Feature #25 polish may add a `MovementBufferPolicy` resource if playtest demands it; the seam is one filter swap. Do NOT add a `MovementLocked` flag — the absence of `MovementAnimation` IS the lock.

- **`MovementAnimation` is ONE component with both translation and rotation fields.** Storage: `from_translation, to_translation, from_rotation, to_rotation, elapsed_secs, duration_secs`. Constructors: `MovementAnimation::translate(...)` (translation move, rotation stays fixed) and `MovementAnimation::rotate(...)` (turn-only, translation stays fixed). **Why:** master research §Pattern 4 has one combined component; two-component split adds the question "which is queried by `Without<...>`?" Wasted bytes for turn-only animations (translation fields unused) are negligible. **How to apply:** new animation kinds (e.g., #25 polish for camera bobbing, #9 atmosphere for breathing) get NEW component types — do not retrofit them onto `MovementAnimation`.

- **Animation pattern mirrors `FadeIn`/`FadeOut` in `src/plugins/audio/bgm.rs:36-67`.** Component-marker carrying `elapsed_secs + duration_secs`; `animate_movement` system ticks `time.delta_secs()`, applies smoothstep + lerp/slerp, removes the component when `elapsed >= duration`. Snap exact at completion to avoid float drift. **How to apply:** new tween animations in later features should follow the same shape — component-marker, `time.delta_secs()` (NOT `delta_seconds()`), exact-snap on completion, `commands.entity(e).remove::<...>()` on finish.

- **System-level state gating, NOT plugin-level.** `Plugin::run_if` does not exist (no `run_if` method on the Plugin trait). `handle_dungeon_input` gates on `in_state(GameState::Dungeon).and(in_state(DungeonSubState::Exploring))` so opening inventory pauses input. `animate_movement` gates only on `in_state(GameState::Dungeon)` so the tween completes if user opens inventory mid-step. **How to apply:** Feature #15 (combat input) follows the same pattern: `combat_input_system.run_if(in_state(GameState::Combat).and(in_state(CombatPhase::PlayerInput)))`. Animation-style systems that should "finish what they started" gate only on the parent GameState, not the SubState.

- **Test scene scaffolding (3 cubes + ground plane + DirectionalLight) is `TestSceneMarker`-tagged for cleanup-on-#8.** Spawned in `spawn_test_scene` on `OnEnter(GameState::Dungeon)`; despawned in `despawn_dungeon_entities` on `OnExit`. **Why:** without it, manual smoke-testing of movement stares into a black void. Three cubes (not one) provide visual landmarks for verifying camera orientation. **How to apply:** Feature #8 deletes the entire `spawn_test_scene` function, the `TestSceneMarker` component, and the `despawn_test_scene` registration in one PR — no other code references them. The marker is a deliberate cleanup seam.

- **Camera spawn pattern: `(Camera3d::default(), Transform::from_xyz(0.0, EYE_HEIGHT, 0.0), DungeonCamera)` as a CHILD of PlayerParty via `children![...]` macro.** No `Camera3dBundle` (does not exist in Bevy 0.18). The required components (`Camera`, `Projection`) are auto-attached via `#[require(Camera, Projection)]` on `Camera3d`. **How to apply:** future camera-spawning code (Feature #8 atmosphere camera, Feature #15 combat camera) uses the same component-tuple pattern. Never type `Camera3dBundle` — it'll fail with "cannot find type". The roadmap mentions the bundle name; ignore the roadmap on this point.

- **`Time::delta_secs()` not `delta_seconds()`.** Bevy 0.18 renamed this. **How to apply:** every tick system uses `time.delta_secs()`. Druum's existing tick systems at `src/plugins/audio/bgm.rs:141` and the new `animate_movement` follow this. `delta_seconds` does NOT exist in 0.18; will be a compile error if typed.

- **Turn-only actions (TurnLeft/TurnRight) write neither `MovedEvent` nor `SfxRequest::Footstep`.** Translation moves write both. Wall-bumps (`can_move == false`) write neither. **Why:** `MovedEvent` contract is "the player crossed a cell boundary"; rotation isn't stepping; wall-bump SFX is a #25 polish concern requiring a new `SfxKind::WallBump` variant (which would touch frozen sfx.rs). **How to apply:** Feature #13 (cell triggers) keys off `MovedEvent` → fires only on translation. Feature #16 (encounter rolls) keys off `MovedEvent` → fires only on translation. If Feature #25 adds turn-SFX, it adds `SfxKind::Turn` and a new `MovementAnimation`-watching system that fires on `Added<MovementAnimation>` filtered by translation-zero — does NOT modify `handle_dungeon_input`'s contract.

- **Test floor helper duplicated, NOT exposed via `pub(crate)`.** Feature #4's `make_floor` helper is `#[cfg(test)]`-private to `data/dungeon.rs`; #7 duplicates ~20 LOC inline rather than refactoring a frozen file. **How to apply:** future features that need an inline test floor follow the same pattern. The duplication cost is < the cost of cross-module test-helper coupling.
